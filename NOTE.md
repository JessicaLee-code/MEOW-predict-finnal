## 冲高分阶段：概念澄清 + 决策（2026-05-28）

> 课程项目；组内无隐藏集、只能讨论选一个代表交老师，老师隐藏集终判。

**概念澄清**

- **GBDT（梯度提升树）**：串行建树、每棵拟合前面残差。对照面 bagging（ExtraTrees=并行平均、降方差）vs boosting（GBDT=串行补错、抓交互）。**LightGBM/XGBoost 就是 GBDT**——一直在跑、只是没对上名。
- **Ensemble = 融合 = P5**：多模型预测组合，去相关的错相互抵消 → 更准更稳。ExtraTrees 自身是同质集成；P5 要异质集成（Ridge+LightGBM+ExtraTrees，机制/盲区不同）。
- **OOF（折外预测）**：只有二层 stacking 才需要——训融合器要用"模型没见过该样本"的预测，否则作弊过拟合。rolling 每折的验证预测就是天然 OOF。
- **DL 喂"原始"≠ 裸喂**：序列模型输入是 LOB/成交的时间路径，但必须先平稳化（价转收益、量转相对不平衡）+ 归一化（前日/滚动 z-score）+ 防泄漏开窗；归一化是最难最易翻车的一步。真量化两派：特征派（手工因子+GBDT/线性，产线主食）vs 端到端 DL（DeepLOB 类）——端到端不是免费午餐，弱信号/数据有限时常打不过特征+GBDT。
- **小波变换 = 特征（不是模型）**：把价/量序列按频率分解，各频带系数当新列喂 GBDT；便宜、属"新种类"多尺度信息可试，但别指望它单独冲 0.10。

**决策（各一句理由）**

- **稳健 = 竞争武器**：组内无隐藏集靠讨论选代表、老师隐藏集终判 → 冲"可辩护可泛化的分"，不刷 dev。
- **传统留 Mac、DL 上 Windows 4060**：分工互不抢算力；传统这套很轻、Mac 够用。
- **P5 加权平均先行**：零过拟合、可辩护、是"融合值不值"的诚实基线；明显更好才上 stacking。
- **DL 低风险先行**：先"已有特征当序列"喂 LSTM 证序列有料，再升 DeepLOB raw-LOB；05-31 回退点。
- **老师 roadmap 分诊**：SVM 跳（核 SVM 在 276 万行不可行 + 线性 SVM 冗余）/ 朴素 DNN 低（表格输 GBDT）/ LSTM·Transformer = 真新 alpha / 小波 = 便宜特征 / 蒸馏·增量·不确定度 = 推迟。

## 评测体系

### 为什么不能只看单次 val_corr

单次验证集的 corr 高，有两种可能：模型真的好，或者恰好那段数据适合这个模型。
Rolling validation 的意义是用多个独立时间段来区分这两种情况——如果每段都好，才是真的好。

### 三层评测的用途分工

**第一层 Dev Rolling（6~10月内部切折）**
- 唯一可以反复用来调参的数据
- 决策依据是汇总指标（rolling_corr_mean、stability_score、positive_fold_rate），不是某一折
- 可以随意跑，不限次数

**第二层 Review Holdout（11月）**
- 候选模型确定后，用来做一次"体外验证"
- 看 11月结果和 rolling 是否一致
- **不能根据 11月结果回头改参数**，否则 11月也变成了调参集
- 一旦污染，直接跳过，依赖 rolling 结果做决策

**第三层 Final Holdout（12月）**
- 12月是我们手里最靠近"老师隐藏测试集"的数据（时间上最近）
- **只跑一次**，在最终提交方案确定之后
- 结果只用于复盘记录，不影响提交决策
- 看了之后再改模型再重跑 = 等于对着答案调参，彻底失去意义

**污染类比**：期末考卷泄题之后做了一遍，再去"复习"，这次考试的成绩不代表真实水平。

### Dev Rolling 的调参粒度

调参针对的是**所有折的汇总结论**，具体来说：
- 新特征/信号：delta_corr ≥ 0.003 才算有效
- 稳定性：stability_score 不下降
- 鲁棒性：最差折不崩、不制造新的强负折（采纳判定一律以 AGENTS §4.6 为准；注意折高度重叠，别把 positive_fold_rate 当硬阈，理由见下文《别被折数骗了把握》）

### 四个 Rolling Profile 的意义

同样的数据，用四种窗口切法，横向对比：

| profile | 训练窗口 | 验证窗口 | 折数 | 主要看什么 |
|---|---|---|---|---|
| short_8d_2d | 8天 | 2天 | ~19折 | 快车道主力：折最多、最便宜，看"有没有用" |
| medium_20d_5d | 20天 | 5天 | ~16折 | 中间插值，已移出日常（只在一次性标定 alpha 时用） |
| long_40d_5d | 40天 | 5天 | ~6折 | 快车道的长窗口斜率/否决票，折少只读方向 |
| expanding_40d_5d | 40天起扩 | 5天 | ~12折 | 慢车道判官，最像真实考试 |

> 这四个 profile 的最终用法以下文《最终落定：两速评测结构》为准。

**窗口长短没有统一答案**，取决于信号的时效性：
- 信号稳定（结构性）→ 长窗口表现不差于短窗口 → 可信
- 信号时效性强 → 只有短窗口好、长窗口差 → 警惕，可能只是近期有效
- 四个 profile 结果一致 → 信号鲁棒

### 综合分数与 profile 明细的关系

`protocol_stability_score` 的正确定位是**第一道自动筛选器**，不是最终决策依据。

它能做到：快速 reject 明显不行的方案，避免看太多数字。  
它做不到：识别"加权平均好但某 profile 有隐患"的情况。

**必须手动钻入 per-profile 明细的场景：**

- 短窗口好、长窗口差 → 信号可能只是近期噪声，不是结构性规律
- 某 profile 的 `rolling_corr_min` 很差（出现强负 fold）→ 均值正但实盘有大回撤风险
- short 和 expanding 结论相反 → 信号的时效性存疑，对训练历史长度非常敏感
- delta_corr 数值处于边界（+0.003～+0.005）→ 在 105 天数据下置信度不够，需要 profile 一致性来补强

**跨视角一致性比单一数字可靠（最终口径见下文《最终落定：两速评测结构》）：**

> 一个真实的结构性信号不应该对训练窗口长度特别敏感——快车道里 short 与 long 同向、到慢车道 expanding 仍成立，才算稳。具体采纳门槛以 AGENTS §4.6 为准。

**（已废弃）加权综合分数：** 早期用过一套 per-profile 加权综合分（medium 0.35 最高，代表"20 天窗口最有代表性"的先验）。两速结构落定后**不再用加权分当第一排序键**（改为按 protocol_corr_mean 排序、stability 守门，见 AGENTS §4.6），此段仅作存档。

### Dev Rolling 的核心局限

6~10月只有约 105 个交易日，可能只覆盖 1~2 种市场状态。  
Long profile 只能切出 ~6 折，统计上很脆弱。

**各 profile 可信度并不相等：**

| profile | 折数 | 独立性 | 可信程度 |
|---|---|---|---|
| short_8d_2d | ~19折 | 相邻重叠 60%+，实际约 5~6 个独立样本 | 折多但窗口短 |
| medium_20d_5d | ~16折 | 同上 | 中间插值，已移出日常筛选 |
| long_40d_5d | ~6折 | 样本最少，方差极大 | 只用来筛"明确有害"，正向证据弱 |
| expanding_40d_5d | ~12折 | 较好 | 模拟真实历史积累，权重偏低但结论可信 |

**应对方式不是扩大调参集（不能动 11/12月），而是提高门槛和要求一致性：**

- 边界情况（delta_corr ≤ 0.003）不接受，除非多个 profile 一致
- delta_corr ≥ 0.005 且 short 与 expanding 同向为正、long 不显著翻负、每日 IC 也改善 → 有信心（完整门槛见 AGENTS §4.6）
- delta_corr = 0.003～0.005 → 不信，等 11 月 Review Holdout 验证
- Long profile 6 折结果不显著 ≠ 信号无效；6 折结果好看 ≠ 可靠
- Review Holdout（11月）的价值因此更大，是另一个月的真正独立样本

### 为什么 +0.003 门槛必须配合一致性和 Review Holdout

把数字摊开：基线 `rolling_corr_std≈0.015`，short 实际只有 ~5–6 个独立样本，均值标准误 ≈ 0.015/√6 ≈ 0.006。

也就是说，**单看一个 profile，+0.003 比一个标准误还小，根本"看不清"它是真信号还是运气**。这解释了为什么不能只看一个数：

- 真正的裁判是「short 与 expanding 同向、每日 IC 也改善」+ 11 月 Review Holdout（另一个独立月份），而不是 Dev Rolling 上的单点增益
- O/T/C 加起来有 ~20 个 spec 都在对同一份 Dev Rolling 比，纯靠运气也会有几个冲过 +0.003（多重比较 / garden of forking paths）。所以每阶段要先把候选集固定下来一次跑完，不要边加边筛
- **关键澄清（2026-05-27）**：这条地板约束的是"**采不采纳**"（promote / 进 holdout），**不是"要不要进 expanding 验证"**。低于 +0.003 但跨视角一致的干净候选（如 C2/O4 单独）照样该进 expanding 让判官看（宽进严出，AGENTS §4.4）。地板当采纳门槛、不当进 expanding 闸刀。

### expanding 是 promote 的必经关口（但宽进严出，2026-05-27 修正）

老师的隐藏集是时间外推，最像它的训练方式是「用到目前为止的全部历史训练、往前预测」——这正是 `expanding_40d_5d` 做的事。short/long 滑窗虽然便宜稳定，却最不像真实任务。

所以日常用 short+long 快筛（long 提供长窗口斜率/否决票），但**一个信号在 promote 前必须单独跑过 expanding 确认**：在短滑窗上好看、一进"累积全历史"就不灵的信号，恰恰是最危险的那一类。expanding 慢、要单独跑、得配 memory guard，但不能因此跳过（具体提速见《基础设施这次要改什么、为什么》）。

**但要分清两件事，别把"必经关口"读成"高门槛"（2026-05-27 修正）：**
- 「promote 必须先过 expanding」说的是**方向**——想采纳，就得先让 expanding 这个判官看过。这层保留。
- 它**不等于**「只有数字够高（过 +0.003 地板）的候选才配进 expanding」。这是我之前犯的错：C2、O4 单独跨视角一致、小而真，只因 Δ<0.003 就被我挡在 expanding 外，从没让最可信的判官亲自看一眼，最后只让凑过地板的组合 X1 进了。
- 正确口径是**宽进严出**：进 expanding 的准入看的是"有没有真收益迹象"（跨 profile 同向、不伤稳定性），不是单个数字；凡有戏的都该进去验证。能不能采纳，才由 expanding 上站不站得住 + holdout 对不对得上来严判。+0.003 是"采不采纳"的参考线，不是"要不要验证"的闸刀。expanding 要"少看"是靠快车道先滤掉明显有害/零增益的来控次数，不是靠数字地板武断卡。

## 数据 ground truth（2026-05-24 让 agent 核查后落实）

一直没确定的几个事实，让 coding agent 去代码和数据里核实了，作为后面定口径的数值依据。

### fret12 到底是什么

就是简单收益（不是对数收益）：`fret12 = (12 步之后的 midpx − 当前 midpx) / 当前 midpx`。

- interval 是分钟级，交易时段是上午 9:30–11:30、下午 13:00–14:45（每天约 226 个，注意收在 14:45，不是 15:00）。
- 不跨夜。但有个细节：临近 11:30 的那十几个 interval，它们的"12 步之后"其实跨过了 90 分钟午休——这个老师已经替我们算好写进数据了，不用自己处理。同样，临近 14:45 的最后十几个 interval，"12 步之后"落在尾盘/集合竞价那段，也是另一种性质的收益。

### fret12 的分布长什么样（以 20230601 为例，约 6.9 万行）

- 均值 +0.00064（几乎为 0，轻微正偏，不用专门去中心化），标准差 0.0072。
- **偏度 1.87、峰度约 15——尾巴很厚**；最小 / 最大 = −5.85% / +9.08%。
- **没有涨跌停（±10%）造成的截断或堆积**；超过 5 个标准差的极端值约占 0.5%。
- 每个时间截面上大约 **300 只股票**（6.9 万 ÷ 226 个 interval ≈ 307）。

### 为什么训练时要把目标"削顶"（winsorize）

尾巴厚意味着：用 MSE 训练时，少数暴涨暴跌的样本因为误差被平方，会对 Ridge 的系数产生过大影响，把模型往这些极端点上拽。可老师评分用的是相关性，关心的是大多数样本排得对不对，不是把极端点拟合得多准。所以把训练目标削顶（按训练集分位裁两边、只裁训练标签、最终提交永远是原始 `fret12`），能让系数更贴合主体、更稳。

要纠正一个直觉误区：削顶**不是**"对评分没影响、只是让梯度好看"。它不改评分的算法，但它改了训练出来的系数，系数变了预测就变了，最终的相关性自然会变。到底是好是坏要靠 rolling 实测（极端值是真信号还是噪声，事先不知道），所以默认带上、但以实测为准。操作规则写在 AGENTS §7.11。

### 午休/收盘这两段边界，要不要专门处理？——别钻牛角尖

这两段加起来约占 10% 的行，确实是另一种收益、是极端值的来源之一。但它对最终分数是二阶影响，而且**削顶这个更通用的招已经覆盖了大半**（不管极端值是哪来的都压住）。所以正确做法是：当成一个候选，加一个"是否边界时段"的标记特征，跑一次 rolling 看有没有用（归到 P3.5），有用就留、没用就扔。**先削顶，如果之后发现问题还是集中在这些边界折，再补这个标记。** 不要再往下做理论推演了，那就是钻牛角尖。

### 现在的特征和泄漏情况

- 9 个 stage 合起来约 **461–462 个特征**（菜单很大，但 R02 基线只用了其中一部分）。
- 每天每个 stage 都没有缺失列、没有常数列，**只有 regime 的两列例外**（`state_spread_cs` / `state_activity_cs`）：它们是"一天一个值"的广播量，对"选哪只股票"几乎没用，只能蹭一点大盘当天的方向；以后上树模型（P4）时它们会造很多重复分裂，要留意。
- 主路径**没有结构性泄漏**（StandardScaler 只在训练集上 fit）。唯一要在看 Final Holdout 之前确认的边界，是 `choose_postprocess_params` 这个在验证集上调参的函数，它的参考集合要钉死。
- 提交契约：预测列必须叫 `forecast`、每个 (symbol,date,interval) 给一个有限值、行的顺序要和 `genFeatures` 的输出对齐（详见 AGENTS §十）。

## 评测协议的本轮收敛（2026-05-24）

### 阈值是地板，不是采纳标准

`0.003~0.005` 只是一条"低于它就别想直接采纳"的安全地板。单个 profile 上 +0.003，比一个标准误（≈0.006）还小，光看它分不清是真信号还是运气。真正决定采不采纳的，是这点增益有没有被几个互相独立的角度同时印证：short / long / expanding 同不同向、每日 IC 有没有也改善、11 月再看一次对不对得上。测试集完全未知，所以宁可保守——这条不算太严。

### 把"每日 IC"提上来当一等诊断：分清你赚的是"大盘的钱"还是"选股的钱"

老师是把所有预测拼起来算一个相关性，所以"猜对某天/某段整体涨跌"（公共分量）也是算分的。问题是：靠猜大盘方向赚的分，换一种行情就没了，脆；靠在同一时刻把股票排序排对赚的分，才稳。怎么分辨？看池化相关性和每日截面 IC 的变化是否一致：池化相关性涨了、但每日 IC 没涨，说明这点提升多半来自猜大盘方向；每日 IC 也跟着涨，才是真的选股能力变强。代码里其实已经在算每日 IC 了，只是没拿来当主力——把它的"均值 ÷ 波动"提上来、和池化相关性并列看，这是零成本的。

### 别被折数骗了把握

short 切出 19 折，但相邻的折训练数据重叠 60% 以上，实际只相当于 5~6 个独立样本。所以"19 折里 16 折为正"听着很硬，打完折其实只是"5~6 块里 4~5 块为正"，没那么硬。结论：别太信折数和正折率，多靠那条约 105 天、独立性强得多的每日 IC 序列。

### 协议本身：是对的，但有冗余；优化方向是"减和聚焦"，不是"加"

四个 profile 没泄漏、切分也合法，设计是对的。但 short / medium / long 这三个，主要都在测同一件事——"你需要多少训练历史"，信息重复；只有 expanding（用到目前为止的全部历史、往前预测）最像老师那场真实考试。所以优化方向不是拆掉重建（那是折腾），而是把注意力重心挪过去：复核一个候选时以 expanding + 每日 IC 为主，short / long 当佐证。**现阶段不要新增 profile**（比如按行情分段那种），那是没必要的复杂度，也是钻牛角尖。具体怎么逐 profile 读，见 AGENTS §4.7。

### 最终落定：两速评测结构（2026-05-25）

上面说"复核以 expanding 为主"，这次把它彻底想清楚、落成一条可执行的两速结构：

- **快车道（每个候选都跑，~10–15 min）：short + long + 每日 IC。** short 折最多、最便宜，负责"这特征到底有没有用"；long 负责"训练历史拉长后还灵不灵"——short 上涨、long 衰减或翻负，就是最危险的"只靠近期"信号，当场抓住。medium 是 short 和 long 中间的冗余插值，移出日常（只在一次性标定 alpha 时用它的折数）。
- **慢车道（每批候选只跑一次）：expanding + 每日 IC 拆解 + 11 月 Review。** expanding 最像老师那场考试，但它慢、且**同一窗口看多了会被过拟合、不再可信**，所以只在"要不要并入 backbone"这种不可逆决策上用，少看。
- **为什么不把 expanding 塞进每一轮**：一是贵（每轮都付不起），二是会污染这个最可信的判官。"见效快但靠近期"这种坑，short→long 的斜率在快车道就能便宜抓到，不必每轮付 expanding 的钱。
- 各阶段（P1–P5）分别用哪条车道、看什么诊断，见 AGENTS §4.4。

## 基础设施这次要改什么、为什么（2026-05-25）

> 这几条是评测口径定下来后、连带要让 coding agent 改的工程项。我和复审 agent 都不碰代码，具体实施规格写在 `docs/specs/开跑前编码指导_评测口径与提速.md`，这里只记"为什么 / 改了什么"，供我回来时查看。

**背景**：expanding 这次串行跑了一轮，真实算力约 2.5 小时（中间机器睡了半小时，所以前台显示 2h34）。瓶颈不是内存（串行能跑完），是算力——几乎全花在 R03/R01/R02 三个重 spec 上，因为 expanding 后段训练窗口长到 ~95 天、近千万行，Ridge 攒正规方程（代价 ∝ 行数 × 列数²）一折就要几分钟。

**改什么**：

1. **日常 suite 默认只含 short + long**（去掉 medium）。理由见上面两速结构；medium 保留为可临时单跑，不进默认。
2. **expanding 只在提交关口跑，而且只跑"候选 vs 基线"两个 spec**，不是 R00–R04 五个全跑——这一刀就把关口耗时砍掉大半。
3. **expanding 提速（纯执行层，不改协议）**：特征 dtype 改 float32（内存砍半、BLAS 更快，需校验 corr 与 float64 一致）；expanding 批次 worker 提到 2，且 group 按成本均衡切分（别把长窗口重折都堆在一个 worker）。合起来一次关口约 30 min。
4. **明确不做"增量正规方程"那种大改**：它能把 expanding 压到分钟级，但只对线性闭式解有效、P4 上树就失效，而 expanding 又是偶发关口，投入产出不划算。先用上面这些便宜招。
5. **每日 IC、以及它的"均值 ÷ 波动"提为一等输出**：代码已经在算，只是没并进 leaderboard 主视图，补上即可，零额外成本。
6. **make_decision 落成硬契约 + 单测、训练目标 winsorize**：见 AGENTS §4.6 / §7.11。
7. **长跑别让 Mac 睡**：跑前挂 `caffeinate -i -s`，省得中途停半小时、计时也干净。

**为什么现在改**：P1 一开跑就是几十轮换特征。如果每轮都跑全量 + expanding，时间根本耗不起；先把"快车道筛、慢车道少看"这条工程化，才扛得住后面的迭代量。

### `#17` 内存压力边界（2026-05-25）

这次把 `expanding_40d_5d` 按成本均衡切成了两个 heavy group：

- `g0`: fold `0/3/4/7/8/11`，训练窗 `40/55/60/75/80/95`
- `g1`: fold `1/2/5/6/9/10`，训练窗 `45/50/65/70/85/90`

也就是说，真要做 `#17` 的完整 2-worker 验收，最危险的时刻不是“两个轻 fold 并行”，而是**尾段近似 `95 + 90 ≈ 185` 个训练日的两份大训练矩阵同时挂在内存里**。

- 日志：`logs/memory_guard_20260525_d17_max_group.log`
- 单组峰值 RSS：**`8.34 GB`**

这个数字已经足够说明问题：

- 在当前 16GB Mac 上，单个最重 group 就能摸到 `8.34 GB`

- 这不是说“Mac 什么都跑不了”

- **推后的是 `#17` 的最高压力完整验收，不是整个两速评测结构**
- 后续如果换到 32GB 级别机器，再重开这个完整验收最合理

这轮真正把“标准 ridge 用什么 alpha、训练标签默认裁不裁、裁到哪”跑实了。实验约束严格按 AGENTS：

- 只跑 **R02 baseline**
- 只跑 **short + medium**
- `alpha ∈ {0.5, 1, 2, 5, 10, 20}`

三档里最好的 `protocol_corr_mean`：

- `P1/P99`：约 **`0.054181`**
- `P0.5/P99.5`：约 **`0.054123`**
- `不裁`：约 **`0.053483`**

- 不裁明显最差，不值得再犹豫
- 裁是有收益的
- 在“裁”的两个候选里，`P1/P99` 比 `P0.5/P99.5` 更好，而且 `stability` 也不差

- `enabled=True`
- `lower_quantile=0.01`

- 这一步没必要再为了“找绝对最优 alpha”继续折腾
- 平台区里取一个中部值最稳妥
- 之前就用的 `2.0` 没有理由改掉

所以标准 ridge 继续锁：

- `alpha=2.0`

这不是某个 quick/单折的巧合，而是：

- 用了 short + medium 两个 profile
- 全量 folds
- 同一套特征、同一个 baseline、同一条 `float32` 路径

足够支撑把默认值固定下来，后续 P1–P3 不再逐 spec 摇这个旋钮。

## Target Mode 口径

### 为什么现在就要拍板

target mode 不是小实现细节，而是“模型到底在学什么”的定义。  
如果这一层不先收敛，后面很容易出现这样的假提升：

- 训练/验证 corr 看起来更高
- 但最终输出不是老师要的原始 `fret12`
- 或者需要用测试集真实 y 的统计量才能还原

这种提升不能交付，继续投入时间只会把实验主线带偏。

### 这次讨论后的核心结论

**唯一硬标准：最终提交预测值必须对应原始 `fret12` 尺度。**

因此 target mode 不看“是否学得更轻松”，先看“能否无泄漏地恢复原始口径”。

### 允许继续推进的 target 路线

- `raw`
  - 最直接，训练和提交口径一致
- `interval_residual`
  - 只要公共基线在测试时可由已知信息构造，并且预测阶段显式加回，最终仍是 raw `fret12`
- `common_residual`
  - common 分支 + residual 分支最终显式重构 raw `fret12`
- `soft_regime`
  - 可以内部使用变换目标做子模型，但最终 meta 输出必须重新对齐 raw `fret12`

### 放弃的 target 路线

- `date_demean`
- `interval_demean`
- `rank target`
- `quantile target`
- 任何最终仍停留在 residual / demean / rank / zscore 尺度的目标

这些路线的问题不是“学不到东西”，而是**不能直接交付**。  
尤其 `date_demean / interval_demean`：测试时不知道真实 y 的组内均值，没法无泄漏地精确加回。

### 后续新增 target mode 的准入问题

以后看到新 target 方案，先只问一句：

> 测试时能不能只依赖已知特征、已知 meta、训练期固定参数，把预测值恢复成原始 `fret12`？

- 能：可以进正式 rolling
- 不能：最多做研究分支，不进最终提交候选

## 阶段推进与准入标准

### 为什么 P1-P5 要分阶段，而不是全部一起试

因为老师测试集隐藏，内部验证最怕的不是“模型不够复杂”，而是：

- 先把太多自由度混在一起，最后不知道什么真的有效
- 弱信号被复杂模型或融合“化妆”
- 某段时间上看起来更高分，但跨窗口不稳定

所以这条路线的真正目的不是保守本身，而是：

> 先确认 alpha 来自哪里，再确认模型是否真的能更好地利用它。

### 关于“是不是太保守”

如果永远只改单变量，确实会太保守，也可能错过“组合后才出现”的信号。  
这次讨论后的收敛不是“只许单变量”，而是：

- 先单变量或小增量筛选信号族
- 再做少量、有先验的组合放大

也就是：

1. `P1-P3`：先看 OFI / trade impact / conditional momentum 各自有没有独立价值
2. `P3.5`：只补少量高先验交互项
3. `P4`：再比较传统模型
4. `P5`：最后做受限融合

这个顺序的好处是：既不完全保守，也避免组合爆炸。

### 为什么要设 P4 / P5 准入标准

准入标准的存在，是为了防三件事：

1. **防止弱信号被复杂模型“化妆”**
   - 有些特征单独不稳，但上树模型、融合、后处理后会在少数窗口里变好看
   - 如果不先设门槛，很容易把噪声错当成 alpha

2. **节省实验预算**
   - 真正贵的是长 rolling、结果分析、复盘和排错
   - 不是每个候选都值得进入模型大战或融合池

3. **把泛化性前置**
   - 面对隐藏测试集，最重要的不是“在某次验证里最高”
   - 而是“跨 profile 一致、最差折不崩、方向不摇摆”

### 当前认可的阶段边界

- `P1-P3`：默认 `Ridge + raw`，重点是换特征定义，不是调超参数
- `P3.5`：允许少量交互项冲刺，但只做高先验候选，不开大网格
- `P4`：才正式允许传统模型的小范围调参
- `P5`：只融合已经被证明独立有效的分支
- Final Holdout 前：必须做一次交付演练，而不是边提交边补规则

### 当前认可的高先验交互项

- `ofi_total x spread`
- `ofi_total x trade_activity`
- `trade_pressure_qty x spread`
- `lagret12 x ofi_total`
- `lagret12 x order_pressure`
- `trade_pressure_qty x regime_score`

这些项的共同点不是“保证有效”，而是：

- 都有明确微观结构解释
- 更可能体现“某信号只在某种流动性 / 状态下有效”
- 比直接上更复杂模型更便宜、更容易定位原因

### alpha 为什么先扫一次再锁定

alpha 是 Ridge 的 L2 正则强度，不是"哪个实验最优"。当前标准路径写死在 `alpha=2.0`，P0 整套基线都是这个值——但从没扫过，只知道"2.0 给出 corr≈0.048"，不知道它是不是好值。

P1–P3 是"R02 + 一个特征组"的单变量对比，每加一个组特征维度都在变。如果 alpha 不固定，就会把两件事混在一起：到底是这个特征有用，还是这个特征集恰好更配某个 alpha？所以先用 R02 扫一遍确认 2.0 在平台区、然后锁定，让 P1–P3 的对比只剩"特征"这一个变量在动。真正的 per-fold 调 alpha 留到 P4 选模型阶段再做。

### 组合不可加：为什么叠加要重跑

单组各自过线，不代表叠起来分数会相加。OFI 和成交冲击都在刻画买卖压力，信息有重叠——第二个组的增益里有一部分第一个已经解释过了，合并后通常小于之和；维度变多在固定 alpha 下还可能拟合噪声，甚至比只加最好的单组更差；偶尔互补会大于之和，但不能预设。结论：组合必须当成新 spec 实际跑一遍，不能用单组 delta 算账。这是单组筛选通向组合 / 融合的强制关卡。

## 特征管道设计宪法

### 核心原则

> 系统不是为了做漂亮的 feature store，而是为了支撑冲高分实验的快速迭代、自由试错、计算复用和特征生命周期管理。

这条原则是在 2026-05-23 的重构设计讨论中确立的。当时 GPT 审计提出了包含 7 个子系统（FeatureComposer、Diagnostics、Ledger 等）的方案，讨论后砍掉了所有过度设计，只保留了三个核心组件：FeatureRegistry（特征定义 + 依赖 DAG）、FeatureStore（磁盘缓存管理）、FeatureLoader（无状态读取）。

### 一票否决规则

以下任一条成立则方案有问题（已写入 AGENTS.md 第二节）：

- 为了试一个新特征，需要手动改多个无关模块
- 改一个特征组导致全量 144 天所有组无脑重算
- 缓存失效规则不清楚，结果不知来自新代码还是旧缓存
- 最终 meow.py 离开本地 cache 就跑不起来

### 背景：为什么走到这一步

起因是 `--suite ridge --n-workers 4` 跑 OOM。排查后发现 ExperimentRunner 里 4 个内存 cache 无上限积累，单 worker 在 long/expanding profile 下峰值 18-20 GB，4 worker 总峰值 >50 GB，16 GB Mac 必然 OOM。

深挖后发现 OOM 只是表象，更根本的问题是：
1. `feat_engine.py` 的 `build()` 是 monolithic 线性链，但实际依赖是 DAG（ofi 完全不依赖 base）
2. 改一个特征组要重算全部 9 组 × 144 天 ≈ 17 分钟
3. `select_groups()` 和 builder 两处信息不同步
4. ExperimentRunner 900 行 god object，缓存/加载/训练/评估全揉一起

所以从"修 OOM"扩展到了"重构特征管道"，趁项目在重启阶段、技术债还少，把基础设施做对。

### 明确不做的事

Hamilton、YAML recipe 系统、表达式 DSL、SQLite manifest、完整生命周期管理平台、column-level 状态管理、DuckDB/PyArrow dataset 优化、跨日 context 抽象、复杂 fingerprint、FeatureComposer。

如果将来 stage 数增长到 30+ 或需要跨日特征，可以重新评估 Hamilton 迁移。

## 特征侧验证的战略发现（2026-05-26，跑完 P1+P2 后）

这是第一次拿真实验结果验证"往基线上加新特征族"这条路，结论比较硬，值得单独记下来，因为它直接改变了我对后面几步的预期。

**一句话：在当前线性 Ridge 基线上，单独追加一个"和基线信息重叠"的特征族，基本捞不到分，甚至会伤稳定性。**

具体怎么发现的：

- **OFI（P1）和成交冲击（P2）两族，全军覆没。** OFI 最好的候选只比基线高 0.0016，成交冲击最好的高 0.0010，都没摸到 +0.003 这条地板（这条地板本身还只有半个标准误那么大，§4.3）。
- **根因是同一个，而且不显眼**：我们的基线 R02 里有个 `norm_core` 特征组，它早就把 OFI 和成交冲击做了"横截面标准化"的版本塞进去了（`ofi_*_cs_z`、`trade_pressure_*_cs_z` 这些）。也就是说，这两族信号里**最稳、最经得起换时间段的那部分，基线已经吃掉了**。我们 P1/P2 再加的，无非是同族的原始版、动态版、交互版——对线性模型来说要么是重复信息，要么是噪声。
- **更要警惕的是成交冲击那个"虚涨真伤"的坑**：T1 的总分（池化 corr）看起来还涨了 +0.0010，但拆开每折一看，它在 short 窗口造出了一个 −0.040 的强负折，全折正率从 100% 掉到 84%，最坏折直接由正翻负。**总分是被 long 窗口那点涨幅拽上去的，代价是最坏情况变烂了。** 这正是为什么不能只看一个总分/一个 decision 标签就下结论——必须钻进每折去看（§4.7）。

**这对后面的预期意味着什么：**

1. **P3（条件动量）之前先把期望调低。** 只要还是"基线 + 同族线性追加"这个套路，大概率还是难破地板。条件动量稍微不同的是它带了 lagret×ofi、lagret×spread 这类交互项，理论上是基线没显式建过的乘积信息——但线性 Ridge 能不能利用乘积，P2 的 T3 交互项（零增益）已经敲了警钟，别抱太大希望。
2. **真正可能有增量的方向，更可能是这两类**：① 基线还没碰过的**全新信号族**（不是 OFI/成交冲击的变体）；② 需要**非线性模型**才能榨出来的交互（这要等 P4 上树才验证，现在线性下看不出来）。
3. **所以队列后面的 alpha 敏感性（Q2）很关键**：它要回答"O4/T1 那点微弱的正信号，是不是被 alpha=2 这个正则强度给压住了"。如果放松正则（alpha=5/20）后它们能冒头，说明信号是真的、只是被埋了；如果还是不动，那就坐实"基线已吃干净"。

> 这条发现的价值在于：它把"无脑加特征"这条直觉路堵死了，让我们把有限的自由度花在更可能有回报的地方（新族 / 非线性 / 真交互），而不是在已被 cross-z 吃掉的同族里反复试。

## P1–Q4 特征侧收官 + 对 P4 的建议（2026-05-26 交回点）

整个特征侧穷尽跑完了（P1 OFI、P2 成交冲击、P3 条件动量、Q2 alpha、Q3 P3.5 交互、Q4 跨族组合）。一句话总结：**单族线性加特征基本死路，但把两个"小而真"的信号叠起来，第一次也是唯一一次真正破了地板——这就是 X1。**

### 跑出来的全景（按"有没有用"排）

| 试了什么 | 结果 | 为什么 |
|---|---|---|
| P1 OFI 单族（6 个变体） | 全否，最好 +0.0016 | 基线 norm_core 已含横截面标准化 OFI，稳健部分被吃掉 |
| P2 成交冲击单族 | 全否，T1"虚涨真伤"（总分涨但 short 造 −0.04 强负折） | 同上；原始/动态版只加噪、伤最坏折 |
| Q2 alpha=5/20 | 否，alpha 不是杠杆 | O4 跨 10× alpha 纹丝不动；T1 的 short 崩塌加正则也救不了（特征本身在小样本抓伪相关） |
| Q3 §7.10 新交互（4 个） | 全否，零增益 | OFI×流动性、成交压力×regime 这些乘积线性下没独立信息 |
| **C2 动量×微结构交互** | 小而真 +0.0019（低于地板） | lagret×{ofi,spread,...} 跨多窗口，是基线没显式建过的乘积信号 |
| **O4 OFI 全集（长窗）** | 小而真 +0.0027（低于地板） | OFI 在长训练窗有干净的结构性残余增量 |
| **Q4 X1 = O4+C2 叠加** | **破地板，三视角全过** | 两个独立小信号几乎可加（+0.0016+0.0019≈实测 +0.00305） |

### 一句话统一根因：三族都死在"基线已含该族主体"（2026-05-27 补）

OFI、成交冲击、**动量/反转**三族单族追加全走弱，根因是同一个、而且藏得深——**每一族最核心、最可外推的那一项，基线 R02 早就含了**：

- OFI / 成交冲击：`norm_core` 里的横截面标准化版本（`ofi_*_cs_z` / `trade_pressure_*_cs_z`）。
- **动量 / 反转：`legacy` 里直接含 `lagret12`**（`LEGACY_COLS`，feature_registry.py）——12 步滞后收益、和预测目标 `fret12` 同窗口，正是动量/反转信号的主体。

所以"动量为什么也不行"这个直觉上的怪点，答案是：**它的主体 `lagret12` 根本不是 P3 才加的，从一开始就在基线里生效**（Ridge 会自己学符号——正系数=动量、负系数=反转）。P3 的"条件动量"测的其实是"在已有 `lagret12` 之上，再加多窗口（lagret1/3/6/24）+变体（_abs/_sign）+交互（lagret×{ofi,spread,...}）"的**边际**，自然小。唯一榨出东西的是 C2 那批**交互**——因为"动量只在某种微结构状态下有效"这个**条件化信息**，是基线线性项里唯一没显式建过的。这也正是 C2 干净、X1 成立的根本理由。

### X1 到底有多硬

X1 = 基线 + `ofi_safe` + `conditional_momentum_interaction`。它在三个独立窗口上的表现：

- short(8天训练)：Δ+0.0025、daily IC 升、最坏折升
- long(40天训练)：Δ+0.0036、6 折全正、最坏折升
- **expanding(累积到约 95 天、最像真实考试)：Δ+0.0044、12 折全正、t=7.35、几乎不过拟合、最坏折升、daily IC 升**

最关键的一点：**增益随训练历史拉长越变越大（0.0025→0.0036→0.0044）**。这跟最危险的"只在近期有效"信号正好相反——它说明用越多历史去预测未来，这个信号越可靠。对一个分布完全未知、可能是时间外推的隐藏测试集，这正是我们最想要的性质。daily IC 在三个窗口都升，说明赚的是"选股的钱"不是"猜大盘方向的钱"。

它现在的标签是 review（+0.0044 还没到 auto-promote 的 0.005），但按我们自己定的规矩（§4.3：真正的关卡是跨视角一致性、不是数字本身），X1 的一致性已经到顶了。

## 转树认识：线性收官、为什么转树、怎么转（2026-05-27）

特征侧（P1–Q4，线性）跑完后想清楚的几件事，决定 P4 怎么开。全量实验明细见 `docs/实验记录.md`，这里只记"为什么"。

### 线性阶段是"立地基"不是"求最优"

线性这一程的产出 ~90% 可复用：评测体系（两速 rolling / make_decision / 看门狗）、数据认知（三族主体被基线吃、厚尾、无泄漏、隐藏集未知）、提交通道、9 stage 特征工程。不可复用的只有"特征在线性上的贡献排序"这一个结论。所以不是白做——只是该收手时在低上限模型上多抠了几下（卡地板、补跑、纠结 X1）。教训：立完地基就该转，别在上限低的模型上追求精细最优。

### 特征价值依赖模型，不是特征固有属性

"某特征有多大用"是"特征 × 模型"的联合属性。我们拿线性（最不会用交互的模型）当了特征价值的唯一裁判，会系统性低估"价值藏在非线性/交互里"的特征——这就是偏颇。但代价可控：特征一个没删（都在 registry），换树能重喂重评。

### 组合爆炸是伪危机——只要不做笛卡尔积

手工一组组试特征子集是线性被逼的（维度多则拟合噪声）。树自带特征选择 + 对无关列鲁棒 → **喂大集让树筛，不搜 2^N 组合**。所以不是"模型数 × 特征组合数"爆炸，而是 ~2 套特征集 × ~5 模型、expanding 只在决赛 2–3 个跑。反偏颇顺带做：树喂大集看重要性，被线性低估的特征若冒头 = 线性错杀，一次回答，不重跑 P1–Q4。

### 树前置认识（树的特征需求和线性相反）

- 树**不需要手工交互**（自建交互）→ C2/X1 那种乘积列对树冗余，可不喂。
- 但树**需要 cross-z/cross-rank**（截面排名）→ 树分裂阈值是跨天全局的、不会自动按天截面归一化，对"选股=截面排序"任务这些列要留、甚至更关键。
- 树对脏列敏感：`regime` 广播量（一天一值）造重复分裂，喂树时清掉。
- winsorize 对树要重评（线性 MSE 防尾部用的，树机制不同）。
- 树路径已接通（`experiment_runner.py` 的 ExtraTrees/HistGB），不用为树改基建、不破坏线性路径。

## 选模型阶段：评测协议为什么要改、怎么改（2026-05-27）

转树前想清楚的一件大事：**我们整套评测协议（两速 rolling、+0.003 地板、make_decision、alpha/winsorize 锁、快车道 short 主筛），全是为"模型固定、特征是变量"设计的。** P1–Q4 每个 spec 都是"R02 + 某特征组"跑 ridge，所以一切围绕"这个特征增量能不能从噪声里分辨出来"标定。现在 P4 把轴翻过来——**特征按模型类型指派、模型类型成了变量**。这一翻，原协议很多东西失效或要改。规则正文落在 AGENTS §4.9，这里只记"为什么"。

### 为什么 +0.003 地板和 make_decision 在选模型时退役

地板 +0.003 是这么来的：dev 窗口 short 实际只有 ~6 个独立样本，基线 corr 的标准误 ≈0.006，所以"特征增量"要到半个标准误以上才勉强能跟运气区分。**但这套推导的对象是"特征增量"**。换模型不是加一列特征，是整个预测函数变了，可能全局 +0.01 也可能 −0.02，地板的统计前提根本不成立。make_decision 同理——它是"候选 vs 固定基线"的 promote/review/reject 分诊，而选模型是**排序挑最优**，没有"基线候选"这个二元关系。所以这两样在选模型语境退役，换成"双镜头排序 + expanding/holdout 确认"。

### 为什么 short 对树不公平、要把重心挪到 long/expanding

short 是 8 天训练窗。ridge 带强正则（alpha=2），8 天也能稳稳给个解；但树吃数据——8 天里一棵树要么数据不够（欠拟合、方差大），要么在小样本上过拟合。**所以拿 short 的分数给模型排序，会系统性冤枉树**，不是因为树差，是因为树被饿着。线性时代 short 是"折最多最便宜"的主筛主力，到选模型这里它反而最不可信。处理办法：short **降级为过拟合诊断**（"short 高、long 低"正好是小窗过拟合的红灯），排序看 long+expanding；便宜初筛改用 long-only（对树公平、又比 expanding 便宜）。

### 为什么判官要"均值 + 鲁棒"两个一起看

这是想了很久才理顺的一点。老师最终只算**一个 pooled Pearson**（把评测窗所有行拼起来算一次相关性），那是个**均值型**的量——所以严格说，我们被打分的东西最接近 `corr_mean`，不是最坏折。那为什么还要看 minimax（最坏折）？因为隐藏集是从一个**完全未知的分布里抽的一次**，而我们的各折是在近似"可能抽到什么行情"；老师那一个 pooled corr，就像从这个分布里**抽中了某一折**，但我们不知道它会像哪一折。优化均值=赌它"长得像平均行情"；minimax=赌它可能是最难的那种、买保险。两者都不能丢：丢了均值就脱离了真实评分口径，丢了鲁棒就在未知分布上裸奔。所以**双镜头同时看、人工权衡**，维持"均值有竞争力 + 最坏折稳"这个原则，不写死成一个公式、也不退化成单指标自动判卷（跟项目一贯的"不自动判卷"是一回事）。目标很明确：**尽快提分**，但分要是经得起未知行情的分。

### 为什么每模型用各自最佳特征集（而不是统一特征集公平比）

本来"公平比模型"应该固定同一套特征。但我们的真实目标不是"评出哪个模型学术上更强"，是"**最终交上去哪套配置最准**"。所以让每个模型用它最擅长的特征集：树喂去交互+清脏列的大集让它自选，线性用 X1。代价是"赢了归功于模型还是特征"说不清——但既然只在乎提交分、不在乎归因，这个混淆可以接受，结论只说"这套配置更准"。而且这么指派**不会引发组合爆炸**：不是每个模型都去搜子集，是按类型直接给定一套。
