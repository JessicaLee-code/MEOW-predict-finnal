## 2. OOM 根因分析（已完成）

### 2.1 直接证据

日志文件：`logs/4worker首跑.txt`

- 11:52-12:22：short_8d_2d、medium_20d_5d 全部完成（train ~140 万行），无异常
- 12:22:47 进入 long_40d_5d_g0，train_shape 翻倍到 277 万行 × 149 列
- 12:25 之后 3 个 worker 同时 abort：long_g0、expanding_g0、expanding_g1
- 主进程未崩，scheduler 正常打印 leaderboard 后退出（short+medium 有结果，long+expanding 全空）

### 2.2 四层 cache 的内存核算

| 缓存 | 内容 | 单条大小 | 被复用几次 | 省了什么 | 省多久 | 结论 |
|---|---|---:|---:|---|---:|---|
| `_split_cache` | concat 后全列大表 | ~2.3 GB | 同折内 4 次 | pd.concat | <1s | **废掉** |
| `_raw_split_cache` | concat 后原始 tick | ~5 GB | 极少 | pd.concat | <1s | **废掉** |
| `_daily_raw_cache` | 单天原始 h5 | ~17 MB | 极少 | h5 读取 | ~0.3s | **废掉**（OS page cache 已做） |
| `_daily_feature_cache` | 单天 featurize 结果 | ~57 MB | 高 | h5 读取 + featurize | **~7s** | **有价值但失控** |

### 2.3 单 worker 峰值估算（long_40d_5d_g0）

```
_daily_feature_cache（7 折跨折累积 ~100 天 × 57 MB）≈ 5.7 GB
_split_cache（当前折 concat 后整表）             ≈ 2.3 GB
_raw_split_cache                                  ≈ 5 GB（如果被触发）
sklearn 训练副本 + Python overhead                ≈ 1-2 GB
───────────────────────────────────────────────────
单 worker 峰值                                    ≈ 14-20 GB
4 worker 总峰值                                   ≈ 50+ GB → OOM
```

### 2.4 PE0 修复为什么不够

commit `f3c9f6a` 修复了 `_split_cache` 的跨折累积（每折后 clear），但 `_daily_feature_cache` 是**故意保留的**（CLAUDE.md 注释"避免重复 IO"）。这个 cache 才是 long/expanding 阶段的内存大头。

### 2.5 残留进程

主进程日志显示正常退出（打了 leaderboard），2.3 GB 残留进程最可能是 OOM kill 后未回收的 spawn worker 孤儿。用户重启电脑后已清除，无法进一步验证。

## 3. 设计讨论核心结论

### 3.1 什么值得缓存

经过逐层分析，确认：

- **唯一值得缓存的是每天的 featurize 结果**（`FeatureBuilder.build()` 的输出）
- 理由：featurize 是唯一的繁重计算（~7s/天），结果完全确定性，可被所有 spec / 折 / profile / run 复用
- h5 读取由 OS page cache 处理（144 天 × 16.5 MB = 2.4 GB，macOS 统一内存轻松全缓存）
- pd.concat 代价 <1s，不值得用 GB 级内存缓存

### 3.2 缓存应存磁盘而非内存

- 内存 dict 缓存导致：多 worker 各持一份 → 内存翻倍；无上限 → OOM；进程退出即丢失 → 跨 run 无法复用
- 磁盘 parquet 缓存：多 worker 共享读取 → 零冗余；不占进程内存 → 无 OOM 风险；持久化 → 跨 run 复用

### 3.3 build() 的假依赖链

代码写成线性链 `base → lag → roll → patch → ofi → trade_impact → cross → cond_mom → regime`，每步传入包含前面所有列的巨型 DataFrame。

但逐方法审计实际引用的列后，真实依赖是 DAG：

```
raw ─────┬── base ─────┬── lag          （只用 base 的 midpx, obi0, trade_imb, spread）
         │             ├── roll         （只用 base 的 mid_ret1_raw, obi0, trade_imb, order_pressure, spread）
         │             ├── patch        （只用 base 的同上 5 列）
         │             └── regime       （只用 base 的 mid_ret1_raw, order_pressure, spread, trade_activity, cxl_imb）
         │
         ├── ofi                        （只用 raw 的 bid/ask/size/turnover，完全不依赖 base）
         │
         ├── trade_impact               （raw + base 的 spread/order_pressure + ofi 的 ofi_total[软依赖]）
         │
         ├── cross                      （base + ofi + trade_impact 的多列做 cs_z/cs_rank）
         │
         └── conditional_momentum       （raw midpx + base mid_ret1_raw + ofi ofi_total[软] + trade_impact trade_pressure_qty[软]）
```

关键发现：**ofi 和 base 完全独立**。改 ofi 不需要重算 base/lag/roll/patch/regime。

### 3.4 设计宪法

> 系统不是为了做漂亮的 feature store，而是为了支撑冲高分实验的快速迭代、自由试错、计算复用和特征生命周期管理。

一票否决规则：
- 为了试一个新特征，需要手动改多个无关模块 → 方案有问题
- 改一个特征组导致全量 144 天所有组无脑重算 → 方案有问题
- 缓存失效规则不清楚，结果不知来自新代码还是旧缓存 → 方案有问题
- 最终 meow.py 离开本地 cache 就跑不起来 → 方案有问题

### 3.5 GPT 审计与纠偏

GPT 提出了 7 个子系统（Registry / Recipe / Cache / Composer / Pipeline / Diagnostics / Ledger）和 Feature Lifecycle 状态机等设计。用户纠偏后的判断：

**采纳的**：
- 设计宪法和一票否决规则
- 特征生命周期概念（scratch → candidate → promoted → archived）
- meow.py 必须脱离缓存独立运行

**砍掉的过度设计**：
- Recipe YAML 系统 → ALL_SPECS 的 dict 列表已经是 recipe
- Expression Composer → pandas 一行代码就行，不需要表达式引擎
- Lifecycle 状态机 → 注释或 dict 字段够了，9 个组不需要正式状态管理
- 7 个命名子系统 → Diagnostics 已在 eval_protocol.py 里，Ledger 已在 fold_metrics.csv + leaderboard.csv 里

### 3.6 外部框架评估

评估了 Hamilton、DVC、Kedro、Dagster 等。结论：

- Hamilton 最接近需求（函数级 DAG + 物化），但需要大幅重写特征代码适配其范式
- 9 个 stage、144 天数据、单机单人 → 自定义 ~150 行 registry 更快更可控
- 框架的价值在规模化和团队协作，当前不需要
- 如果将来 stage 数增长到 30+ 或需要跨日特征，可再评估 Hamilton 迁移

## 4. 最终方案

### 4.1 改造范围

| 文件 | 操作 | 说明 |
|---|---|---|
| `src/feature_registry.py` | **新建** | FeatureRegistry 类 + 9 个 stage builder 函数 + select_groups 自动派生 |
| `src/feature_store.py` | **新建** | FeatureStore 类（磁盘管理 + 脏检测 + 拓扑重建）+ `__main__` CLI |
| `src/feature_loader.py` | **新建** | FeatureLoader 类（无状态读取 parquet + 选列 + concat） |
| `src/experiment_runner.py` | **修改** | 删 4 个 cache dict + load 系列方法，保留模型训练评估 |
| `src/scheduler.py` | **修改** | worker 改用 FeatureLoader，删 cache 清理 |
| `src/feat_engine.py` | **归档** | builder 逻辑迁入 feature_registry.py，旧文件移至 .archive/ |
| `src/eval_protocol.py` | **微调** | 运行时复制 manifest 快照到结果目录 |
| `experiments/p0_eval_protocol.py` | **微调** | 传入 feature_dir 参数 |

不变：trainer.py、eval_protocol.py 核心逻辑、ALL_SPECS、results/ 已有结果。

### 4.2 磁盘结构

```
data/features/
    manifest.json           ← {stage: {code_hash, built_at, n_days, n_columns}}
    base/20230601.parquet
    base/20230602.parquet
    ...
    ofi/20230601.parquet
    ...
```

### 4.3 预期效果

| 指标 | 旧管道 | 新管道 |
|---|---|---|
| --suite ridge 内存峰值 | >50 GB（OOM） | ≤16 GB |
| 全量特征构建 | N/A（按需计算） | ~17 min（一次性） |
| 改一组后增量重建 | 17 min（全量） | ~5-8 min |
| 加新特征组需改几个文件 | 2 处（builder + select_groups） | 1 处（registry 一个装饰器） |

### 4.4 关键约束

- meow.py 最终提交入口不依赖 data/features/，从 raw h5 实时调 builder 函数构建
- 缓存是开发加速层，builder 函数是真相源，两条路径共享同一份代码

## 7. 代码层面的确认事项

以下是代码审计中确认的事实，实施时无需重新验证：

| 事项 | 结论 | 依据 |
|---|---|---|
| `_build_sequence_features` | dead code，无人调用 | grep 确认只有定义，无调用点 |
| `load_raw_split` / `_daily_raw_cache` | 无外部调用 | 仅在 runner 定义和 scheduler cache 清理中出现 |
| `_split_cache` key 结构 | `(tuple(dates), max_days)`，不含 spec | 同折内 5 spec 共享一条 entry |
| ofi 不依赖 base | `_add_ofi_features` 只引用 raw 的 bid/ask/size/turnover 列 | 逐行审计确认 |
| trade_impact 对 ofi 是软依赖 | `df.get("ofi_total", 0.0)` | 重构后改为硬依赖 |
| 所有特征均为日内计算 | 全部 groupby 含 `"date"` | per-day 构建假设成立 |
| h5 单文件大小 | ~16.5 MB，共 144 天 | ls -lh 确认 |
| featurize 单天耗时 | ~7 秒（4 worker 并行 IO 竞争下） | 日志时间戳推算 |
| `common_residual` 调 `load_feature_split` 2 次 | 加载 train + val | 代码行 703-704 |
| `soft_regime` 调 `load_feature_split` 3 次 | 全列 + 两次 `run_with_groups` 内部各调一次 | 代码行 751-752, 754-769 |
