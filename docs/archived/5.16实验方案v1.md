日期：2026-05-16  
项目：MEOW 金融时序预测  
目标读者：执行实验的 AI / 队友  
版本定位：在当前正式 rolling 只有约 `0.048` 的情况下，用最小实验成本判断是否仍有现实机会冲击 `Pearson Corr > 0.10`。

---

## 0. 当前事实与方案定位

### 0.1 当前实验事实

截至 2026-05-16，目前正式 rolling 口径下最关键的事实是：

| 事实 | 结论 |
|---|---|
| 当前最好 Ridge 增强为 `R02_ridge_legacy_plus_norm_core` | `rolling_corr_mean ≈ 0.047976`，`stability_score ≈ 0.037494` |
| `B0_ridge_legacy` 是当前最稳 backbone | 复杂模块没有稳定压过它 |
| `B6_common_residual` 有一定均值收益 | 但波动较大，只能作为小权重 expert |
| `B7_soft_regime` 有局部收益 | 但稳定性不足，不能作为主线 |
| `patch_summary / cross_rank / residual target / postprocess` | 暂未形成稳定增益 |
| 当前结果离 `0.10` 很远 | 不能继续靠小修小补幻想突破 |

### 0.2 当前核心判断

当前最大问题不是“模型还不够复杂”，而是：

```text
现有信号族的上限没有打开。
```

因此，下一阶段不能继续盲目扩复杂模型，而要优先验证两个问题：

```text
1. 当前正式 rolling 分数低，是否是因为训练窗口 max_train_days=4 太短？
2. 如果不是训练窗口问题，是否可以通过动态订单流 OFI、成交冲击、条件动量/反转打开新信号上限？
```

### 0.3 当前阶段目标

当前阶段不要把目标写成“马上超过 10%”。

当前阶段目标是：

```text
P0: 判断 max_train_days 是否压低了模型表现；
P1: 验证 OFI 动态订单流是否能把 rolling_corr_mean 从约 0.048 抬到 0.055+；
P2: 如果 OFI 有效，再验证 trade impact 和 conditional momentum；
P3: 只有新信号稳定有效后，才做浅模型和受限融合；
P4: 如果所有新信号仍停在 0.05 左右，则停止冲 10%，转向稳健提交和报告加分。
```

## 1. 全局硬约束

以下规则对所有实验生效。

### 1.1 MUST：必须遵守

```text
MUST:
1. 最终预测目标始终是原始 fret12。
2. 所有模型必须同时记录 corr / mse / r2。
3. 所有实验必须使用正式 rolling validation。
4. 所有 rolling split 必须使用同一套切分逻辑，便于公平比较。
5. train 与 validation 之间必须设置 embargo。
6. 所有 rolling / EMA / zscore / rank 特征只能使用当前及历史特征。
7. validation/test 的真实 y 不允许参与特征构造、demean、rank、common component、融合权重选择或后处理调参。
8. 每个实验必须落盘结果表，不能只在终端打印。
9. 每次只新增一类特征组，避免一次性全加导致无法解释。
10. 每个保留实验必须给出保留原因。
```

### 1.2 FORBIDDEN：当前阶段禁止事项

```text
FORBIDDEN:
1. 不要继续扩 soft regime。
2. 不要继续扩 residual target 变体。
3. 不要继续做复杂 postprocess 网格。
4. 不要做 rank blend / group neutralization。
5. 不要上 MLPLOB / Transformer / LSTM / PatchTST。
6. 不要自由 stacking。
7. 不要只看单次 holdout 决策。
8. 不要为了 corr 牺牲 MSE / R² 到明显崩坏。
9. 不要用 final holdout 反复调参。
10. 不要把任何不稳定 expert 作为最终主模型。
```

### 1.3 SHOULD：建议但可根据计算资源调整

```text
SHOULD:
1. 优先使用 Ridge / ElasticNet / HuberRegressor 验证新信号。
2. 只有新信号在线性模型里有效后，才尝试浅树模型。
3. 每个新信号组先跑最小版本，再跑衍生版本。
4. 每个阶段结束后生成阶段总结，说明继续或停止的理由。
```

## 2. 统一验证协议

### 2.1 Rolling split 设置

默认正式 rolling：

```text
n_folds = 5
val_window = 2 days
step = 10 days
embargo >= 12 intervals
```

当前必须重点测试 `max_train_days`，所以初始阶段不要固定为 4 天。

### 2.2 Embargo 规则

因为任务预测未来 12 分钟收益 `fret12`，因此：

```text
embargo >= 12 intervals
```

如果某些特征使用更长的历史 lookback，例如 60 intervals：

```text
embargo = max(12, max_feature_lookback)
```

### 2.3 统一指标

每个 fold 必须记录：

```text
corr
mse
r2
daily_corr_mean
daily_corr_std
train_corr
```

整体记录：

```text
rolling_corr_mean
rolling_corr_std
rolling_corr_min
stability_score
rolling_mse_mean
rolling_r2_mean
daily_corr_mean
daily_corr_std
train_corr_mean
```

稳定性指标固定为：

```text
stability_score = rolling_corr_mean - 0.7 * rolling_corr_std
```

### 2.4 模型保留规则

模型进入下一阶段必须满足至少一个条件：

```text
KEEP IF:
1. rolling_corr_mean 提升 >= 0.003 且 stability_score 不下降；
OR
2. stability_score 提升 >= 0.003 且 rolling_corr_min 不下降；
OR
3. MSE / R² 明显改善且 corr 不明显下降。
```

模型必须丢弃：

```text
DROP IF:
1. 只提升单个 fold；
2. rolling_corr_std 明显变大；
3. rolling_corr_min 明显下降；
4. train_corr 明显升高但 val corr 不升；
5. R² 明显恶化；
6. 结果依赖单次 holdout。
```

## 3. 阶段 P0：训练窗口敏感性测试

### 3.1 目的

当前正式 rolling 使用：

```text
max_train_days = 4
```

这可能太短。由于数据有数月分钟级数据，如果模型只看 4 天，可能严重低估稳定模型的能力。

本阶段只回答一个问题：

```text
当前 0.048 左右的上限，是不是被 max_train_days=4 人为压低了？
```

### 3.2 固定模型

只跑当前最好且稳定的 Ridge 增强：

```text
R02_ridge_legacy_plus_norm_core
```

不要在本阶段换模型，不要新增特征。

### 3.3 实验矩阵

```text
P0_W4   = R02, max_train_days = 4
P0_W10  = R02, max_train_days = 10
P0_W20  = R02, max_train_days = 20
P0_W40  = R02, max_train_days = 40
P0_W80  = R02, max_train_days = 80
P0_EXP  = R02, max_train_days = expanding
```

所有其他设置保持不变。

### 3.4 输出表格

| experiment | max_train_days | fold1_corr | fold2_corr | fold3_corr | fold4_corr | fold5_corr | rolling_corr_mean | rolling_corr_std | rolling_corr_min | stability_score | rolling_mse_mean | rolling_r2_mean | train_corr_mean | keep/drop | reason |
|---|---:|---:|---:|---:|---:|---:|---:|---:|---:|---:|---:|---:|---:|---|---|

### 3.5 判断规则

```text
IF max_train_days ∈ {20, 40, 80, expanding} 明显高于 4:
    说明此前正式 rolling 可能低估模型。
    后续所有实验使用表现最好的 max_train_days。

IF 所有长窗口仍然在 0.045 ~ 0.052:
    说明主要问题不是训练窗口，而是信号族上限不足。
    进入 OFI 动态订单流实验。
```

### 3.6 阶段目标线

```text
GOOD:
rolling_corr_mean >= 0.055

VERY_GOOD:
rolling_corr_mean >= 0.060

BAD:
rolling_corr_mean < 0.052 且 stability_score 没有改善
```

## 4. 阶段 P1：OFI 动态订单流特征

### 4.1 目的

当前 Ridge 系数显示，有效信号主要来自：

```text
trade_imb
ob_imb4 / obi4
ob_imb0 / obi0
order_pressure
trade_imbema5
lagret12
```

这些都指向一个事实：

```text
当前有效信号主要来自订单簿供需压力和成交方向压力。
```

所以最值得赌的新信号是 OFI，即 Order Flow Imbalance。它不是静态问“当前盘口谁多”，而是动态问“盘口压力如何变化”。

### 4.2 字段要求

执行 AI 必须先自动识别字段名。

可能字段包括但不限于：

```text
bid_price_0, bid_price_1, ...
ask_price_0, ask_price_1, ...
bid_size_0, bid_size_1, ...
ask_size_0, ask_size_1, ...

或类似：
bidPx0, askPx0, bidQty0, askQty0
bp0, ap0, bv0, av0
```

如果字段名不一致，必须先输出字段映射表：

| logical_name | actual_column | exists |
|---|---|---|
| bid_price_0 | ... | yes/no |
| bid_size_0 | ... | yes/no |
| ask_price_0 | ... | yes/no |
| ask_size_0 | ... | yes/no |

如果缺少价格或 size 字段，不能强行构造 OFI，必须输出失败原因。

### 4.3 OFI 计算规则

对每只股票，按时间升序排序。

对 bid side：

```text
if bid_price_t > bid_price_{t-1}:
    bid_ofi = bid_size_t

if bid_price_t == bid_price_{t-1}:
    bid_ofi = bid_size_t - bid_size_{t-1}

if bid_price_t < bid_price_{t-1}:
    bid_ofi = -bid_size_{t-1}
```

对 ask side：

```text
if ask_price_t < ask_price_{t-1}:
    ask_ofi = -ask_size_t

if ask_price_t == ask_price_{t-1}:
    ask_ofi = -(ask_size_t - ask_size_{t-1})

if ask_price_t > ask_price_{t-1}:
    ask_ofi = ask_size_{t-1}
```

单档 OFI：

```text
ofi_level_k = bid_ofi_k + ask_ofi_k
```

聚合 OFI：

```text
ofi0 = ofi_level_0
ofi4 = sum(ofi_level_0 ... ofi_level_4)
ofi9 = sum(ofi_level_0 ... ofi_level_9)
```

如果只有 5 档，则构造 `ofi0` 和 `ofi4`；如果只有 1 档，则只构造 `ofi0`。

### 4.4 OFI 衍生特征

对 `ofi0 / ofi4 / ofi9` 构造：

```text
ofi_ema_3
ofi_ema_6
ofi_ema_12
ofi_ema_24

ofi_sum_3
ofi_sum_6
ofi_sum_12
ofi_sum_24

ofi_mean_3
ofi_mean_6
ofi_mean_12
ofi_mean_24

ofi_zscore_12
ofi_zscore_24
ofi_zscore_60

ofi_abs
ofi_sign

ofi_div_total_depth
ofi_div_turnover

ofi_cross_rank_date_interval
```

注意：

```text
所有 EMA / rolling sum / zscore 只能使用当前及历史数据。
cross_rank_date_interval 可以使用当前 interval 的横截面特征值，但不能使用当前或未来 y。
```

### 4.5 OFI 实验矩阵

基准模型使用 P0 阶段选出的最佳训练窗口下的 R02。

```text
O1 = R02 + ofi0/ofi4/ofi9
O2 = R02 + ofi_ema_3_6_12_24
O3 = R02 + ofi_sum_3_6_12_24
O4 = R02 + ofi_zscore_12_24_60
O5 = R02 + ofi_cross_rank_date_interval
O6 = R02 + all_safe_ofi_features
```

### 4.6 OFI 保留规则

```text
KEEP:
rolling_corr_mean 提升 >= 0.003
AND stability_score 不下降
AND rolling_corr_min 不下降或只轻微下降

STRONG_KEEP:
rolling_corr_mean >= 0.055

MAINLINE:
rolling_corr_mean >= 0.060
```

```text
DROP:
1. 只某一折提升；
2. all_safe_ofi_features 不如单个 OFI 子组；
3. std 明显变大；
4. min fold 下降明显；
5. R² 明显恶化。
```

### 4.7 OFI 实现检查

如果 OFI 没有任何提升，必须检查：

```text
1. 是否按 stock 分组并按时间排序；
2. bid/ask 价格方向是否写反；
3. ask_ofi 符号是否写反；
4. size 字段是否误用金额字段；
5. level 聚合是否重复或缺失；
6. rolling/EMA 是否使用未来数据；
7. 特征是否被异常值支配，需要 winsorize。
```

## 5. 阶段 P2：成交冲击 Trade Impact

### 5.1 执行条件

只有当 P1 的 OFI 至少满足以下条件之一时才执行：

```text
1. rolling_corr_mean 提升 >= 0.003；
2. stability_score 提升 >= 0.003；
3. OFI 子组在多个 fold 中均有正向贡献。
```

如果 OFI 完全无效，先检查 OFI 实现，不要直接进入 P2。

### 5.2 基础特征

需要识别字段：

```text
tradeBuyQty
tradeSellQty
tradeBuyTurnover
tradeSellTurnover
nTradeBuy
nTradeSell
```

如果字段名不同，先输出字段映射表。

构造：

```text
signed_trade_qty = tradeBuyQty - tradeSellQty
signed_trade_turnover = tradeBuyTurnover - tradeSellTurnover

trade_pressure_qty =
    signed_trade_qty / (tradeBuyQty + tradeSellQty + eps)

trade_pressure_turnover =
    signed_trade_turnover / (tradeBuyTurnover + tradeSellTurnover + eps)

trade_intensity = nTradeBuy + nTradeSell

avg_trade_size =
    (tradeBuyQty + tradeSellQty) / (nTradeBuy + nTradeSell + eps)

avg_trade_turnover =
    (tradeBuyTurnover + tradeSellTurnover) / (nTradeBuy + nTradeSell + eps)
```

### 5.3 衍生特征

```text
trade_pressure_ema_3
trade_pressure_ema_6
trade_pressure_ema_12
trade_pressure_zscore_12
trade_pressure_zscore_24
trade_pressure_cross_rank
trade_intensity_rank
avg_trade_size_rank
```

### 5.4 交互项

```text
trade_pressure * spread
trade_pressure * volatility
trade_pressure * turnover_rank
trade_pressure * ob_imb0
trade_pressure * ob_imb4
trade_pressure * ofi4
trade_pressure * ofi9
```

### 5.5 实验矩阵

基准使用当前 best model，即：

```text
best_after_P1 = R02 或 R02+best_OFI
```

依次跑：

```text
T1 = best_after_P1 + trade_pressure_base
T2 = best_after_P1 + trade_pressure_ema
T3 = best_after_P1 + trade_pressure_rank_zscore
T4 = best_after_P1 + trade_pressure_interactions
T5 = best_after_P1 + all_safe_trade_impact
```

### 5.6 保留规则

同 OFI。

额外规则：

```text
如果交互项导致 std 明显变大，则只保留基础 trade_pressure，不保留交互项。
```

## 6. 阶段 P3：条件动量 / 条件反转

### 6.1 执行条件

只有当 P1 或 P2 产生稳定增益后才执行。

如果 OFI 和 trade impact 都无效，暂不执行 P3。

### 6.2 目的

当前已有 `lagret12` 有效，但它只表示过去收益。短期金融数据中，过去收益可能代表动量，也可能代表反转。模型需要知道：

```text
什么时候 lagret 是动量？
什么时候 lagret 是反转？
```

所以需要条件化动量特征。

### 6.3 基础 lag return

构造：

```text
lagret1
lagret3
lagret6
lagret12
lagret24
```

如果已有 `lagret12`，保留原版本，同时补齐其他窗口。

所有 lagret 都必须只使用当前及历史价格，不允许使用未来价格。

### 6.4 条件交互

```text
lagret1  * trade_pressure
lagret3  * trade_pressure
lagret6  * trade_pressure
lagret12 * trade_pressure

lagret12 * ofi4
lagret12 * ofi9

lagret12 * spread_rank
lagret12 * volatility_rank
lagret12 * turnover_rank
lagret12 * ob_imb4
```

### 6.5 实验矩阵

```text
C1 = best_after_P2 + lagret_multi_window
C2 = best_after_P2 + lagret_trade_pressure_interactions
C3 = best_after_P2 + lagret_ofi_interactions
C4 = best_after_P2 + lagret_liquidity_interactions
C5 = best_after_P2 + all_safe_conditional_momentum
```

### 6.6 保留规则

```text
KEEP:
rolling_corr_mean 提升
OR stability_score 提升

DROP:
1. 只提升单折；
2. std 明显扩大；
3. min fold 明显下降；
4. R² 明显恶化。
```

## 7. 阶段 P4：稳健模型比较

### 7.1 执行条件

只有当前面至少找到一个稳定新特征组后才执行。

如果 P1/P2/P3 都无效，则不要做模型扩展，直接进入总结和失败判断。

### 7.2 模型优先级

优先使用稳健线性模型：

```text
M1 = Ridge(best_features)
M2 = ElasticNet(best_features)
M3 = HuberRegressor(best_features)
```

辅助浅树模型：

```text
M4 = ExtraTrees_shallow(best_features)
M5 = HistGradientBoosting_shallow(best_features)
```

### 7.3 树模型复杂度限制

ExtraTrees：

```text
n_estimators <= 300
max_depth <= 6
min_samples_leaf >= 300
max_features <= 0.5
bootstrap = True
random_state 固定
```

HistGradientBoosting：

```text
max_leaf_nodes <= 31
learning_rate <= 0.05
l2_regularization >= 0.1
early_stopping = True
random_state 固定
```

### 7.4 保留规则

```text
如果浅树 rolling_corr_mean 高但 std 也明显高：
    不保留为主模型，只能作为小权重 expert。

如果浅树没有超过 Ridge：
    不继续调深，不扩树模型。
```

## 8. 阶段 P5：受限融合

### 8.1 执行条件

只有当至少两个 expert 在 rolling 下有稳定正贡献时才执行。

### 8.2 候选 expert

```text
E0 = best_linear_model
E1 = best_OFI_model
E2 = best_trade_impact_model
E3 = best_conditional_momentum_model
E4 = B6_common_residual
E5 = B7_soft_regime
E6 = shallow_tree_best_features
```

不是所有 expert 都必须存在。只有通过前面保留规则的 expert 才能进入融合。

### 8.3 融合形式

```text
final_pred =
w0 * E0
+ w1 * E1
+ w2 * E2
+ w3 * E3
+ w4 * E4
+ w5 * E5
+ w6 * E6
```

### 8.4 权重限制

```text
w0 >= 0.50
w1 + w2 + w3 <= 0.40
w4 <= 0.10
w5 <= 0.10
w6 <= 0.15
sum(w) = 1
```

### 8.5 禁止融合行为

```text
FORBIDDEN:
1. soft regime 权重大于 0.10；
2. common residual 权重大于 0.10；
3. tree expert 权重大于 0.15；
4. 任一不稳定 expert 权重大于 0.10；
5. 用 final holdout 调权重；
6. 融合退化为某个复杂 expert 单独主导。
```

### 8.6 权重选择

权重只允许在 rolling validation 上选择。

建议先用手动网格，不要直接 OOF stacking：

```text
w0 ∈ {0.50, 0.60, 0.70, 0.80}
w1/w2/w3 ∈ {0.00, 0.05, 0.10, 0.15, 0.20}
w4/w5 ∈ {0.00, 0.05, 0.10}
w6 ∈ {0.00, 0.05, 0.10, 0.15}
```

保留最高 `stability_score`，并同时检查 `rolling_corr_mean`、`rolling_corr_min`、`MSE`、`R²`。

## 9. 阶段判断线

### 9.1 继续冲 10 的条件

```text
IF best rolling_corr_mean >= 0.075:
    可以继续尝试受限融合、浅树、轻微校准，目标冲 0.10。

IF best rolling_corr_mean >= 0.085:
    可以进入最终冲刺阶段，但仍然禁止泄漏和过拟合后处理。
```

### 9.2 保守提交的条件

```text
IF P0 + OFI + trade impact + conditional momentum 后，best rolling_corr_mean 仍 < 0.060:
    说明当前数据/信号下冲 10 现实概率很低。
    停止继续堆模型。
    转向稳健提交、报告完整度、创新性叙述和探索过程拿分。
```

### 9.3 最低有效进展线

```text
P0 后：
    希望 >= 0.055
```

## 12. 报告叙述策略

即使最终没有超过 10%，本方案也要为报告保留价值。

报告中可以这样组织：

```text
1. 金融分钟数据噪声高、非平稳，单次 holdout 不可靠。
2. 我们先用 rolling validation 审计复杂模型，发现复杂模块不稳。
3. 我们确认 Ridge legacy 是稳健 backbone。
4. Ridge 系数显示有效信号集中在订单簿压力和成交方向压力。
5. 因此我们没有盲目堆模型，而是转向动态订单流 OFI、成交冲击和条件动量。
6. 所有新信号都通过严格 rolling、embargo 和三指标评估。
7. 最终模型选择兼顾 Pearson Corr、MSE 和 R²。
```

如果 OFI 有效，可以作为重要创新点：

```text
将静态盘口 imbalance 扩展为动态 Order Flow Imbalance，捕捉买卖队列变化、补单、撤单和价格层级移动带来的短期价格压力。
```

## 14. 最终决策规则

```text

    可以继续冲 10。
```

## 15. 给执行 AI 的最后提醒

当前最重要的不是“多跑实验”，而是“跑对实验”。

请不要擅自添加复杂模型。  
请不要用单次 holdout 鼓励错误方向。  
请不要把高 train corr 当作成功。  
请不要为了冲 10 使用任何可能泄漏的处理。  

现在最关键的两个问题是：

```text
1. max_train_days=4 是否严重低估模型？
2. OFI / trade impact / conditional momentum 是否能提供新的强信号？
```

先回答这两个问题，再决定是否继续冲 10。
