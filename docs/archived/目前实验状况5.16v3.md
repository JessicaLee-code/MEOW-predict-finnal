## 3. 两个不同历史分数的原因解释

当前已确认，两个历史数值来自两套不同实验口径：

### 3.1 正式 rolling 结果

来自 [ridge_enhance_results.csv](D:/code/final/ridge_enhance_results.csv)：

- 实验名：`R02_ridge_legacy_plus_norm_core`
- fold 数：5 折
- 指标：
  - `rolling_corr_mean = 0.047976`
  - `rolling_corr_std = 0.014975`
  - `rolling_corr_min = 0.032599`
  - `stability_score = 0.037494`

### 3.2 train window sensitivity quick 结果

来自 [train_window_sensitivity_quick_results.csv](D:/code/final/train_window_sensitivity_quick_results.csv)：

- 实验名：`R02_ridge_legacy_plus_norm_core_max_train_4`
- fold 数：2 折
- 指标：
  - `rolling_corr_mean = 0.039211`
  - `rolling_corr_std = 0.002293`
  - `rolling_corr_min = 0.036918`
  - `stability_score = 0.037606`

### 3.3 当前判断

这两个结果的主要差别不是模型本身，而是评估口径不同：

- 一个是正式 5 折 rolling
- 一个是 quick train-window sensitivity 的 2 折结果

因此当前不能直接得出“`max_train_days=4` 一定比正式 R02 更好”这个结论。  
必须先把 **同一 split、同一 embargo、同一数据范围、同一 seed** 下的 R02 复现跑干净，再讨论训练窗长度。

### 12.2 是否适合直接传远程

可以传，但不建议走普通 Git 代码仓库。

原因：

1. 普通 Git 不适合直接管理 `2.45 GB` 的数据目录
2. 如果是 GitHub 这类平台，单文件和仓库体积都会很快碰到限制
3. 这类 `.h5` 数据更适合作为“随仓库分离的数据资产”，而不是代码版本历史的一部分

### 12.3 当前更合理的迁移方式

推荐把迁移分成两部分：

1. 代码仓库
   - 只同步代码、文档、必要脚本
   - 不带 `archive/`
   - 不带 `.zip`
   - 不带 `.log`
   - 不带大结果产物

2. 数据包
   - 单独传 `archive/` 或单独传一个压缩包
   - 在新环境解压后，保持目录为 `D:/code/final/archive` 的同等结构即可

如果远程环境支持大文件直传，直接传 `archive.zip` 反而最省事。  
如果远程环境网络不稳，优先传 `archive/` 目录本身或用支持断点续传的工具。
