# 原始数据盘点 与「直接建模盘口」诊断报告

> 时间：2026-06-02 夜（无人值守，纯只读诊断 + 写文档，未训练、未改交付/协议/master）
> 触发：老师说**同一份数据、同一题目，有人做到 12%+**；我们卡在 ~8%（pooled Pearson 0.0776）。
> 目的：搞清楚差距卡在哪，**不是直接试方法**，而是先定位漏分点（先诊断、后探索）。

---

## 0. 一句话结论

**原始数据远比我们以为的丰富**：完整 4 档订单簿 + 逐笔成交流 + 挂撤单流 + 226 步日内序列。
**但我们把它压成了 433 个静态汇总数字，而且所有 DL 都只喂这 433 个汇总、从没直接吃过原始盘口。**
最值钱的几块——**挂撤单动态、盘口形状/深度分布、日内全序列**——基本没用上。

> **「无 LOB / 数据锁死 / raw⊆433」是错误前提。** 我之前据此推出的「信息上限 ~0.10」站不住。
> **冲 12% 的真入口 = 直接在原始盘口序列上建模（DeepLOB 式），而不是继续在 433 上抠融合那一个点。**

---

## ✏️ 校正与进展（2026-06-03 上午）

**昨夜一处说法要校正**：「DL 从没喂过 raw、挂撤单全丢」**不准确**。实际上：
- 项目早有 `RawChannelAdapter`（`models/dl_models.py:336`），把 **59 个原始通道**（含挂撤单 `nAddBuy/nCxlBuy…`、深档盘口、成交明细）做了归一化（价→相对 mid、量→log1p、midpx→日内对数收益）喂模型；
- 它只是**当年只被 TCN 用过**，而 TCN `[B,L,C]` 一次一票、**无截面** → 0.009。

**精确的缺口（比昨夜更准）**：前三轮**从没把"好架构"和"好输入"合在一起**——

| | 架构 | 输入 | 分数 |
|---|---|---|---|
| TCN-on-raw | ❌ 无截面 | ✅ raw 59 通道 | 0.009 |
| XSECTION/GRU | ✅ 带截面 | ❌ 433 压缩摘要 | 0.062 |
| **XSECTION_RAW（本次新建）** | ✅ **带截面** | ✅ **raw 59 通道** | **验证中** |

> 下文 §3「挂撤单只压成 4 比率」等仍**对 433/传统特征链成立**（那条链确实欠用 raw）；但"raw 这条线整个没建"是错的——raw 线存在、只是没接好架构。

**进展**：已新建 `XSECTION_RAW` 卡带（截面架构原样复用 + 输入换 RAW_CHANNELS，`config/model_config.py` + `models/dl_models.py`，注册自检 + GPU sanity 通过，已提交 feat）。**正式 fold2 挑战进行中**（`run_id=20260603_xsection_raw_fold2`，全 309 票/15 epoch/3 seed/带 dump-preds），看能否超过传统 fold2 的 **0.0904**：超过=实锤瓶颈是输入压缩、raw 有肉；不超过=查归一化/通道/架构再铺开。

## 1. 评分口径（已确认，关掉一个台阶）

- 交付评测器 `meow/eval.py:13-18` 与内部 `src/experiment_runner.py:350-357` **双双确认**：头号指标 = **pooled Pearson**（全样本 `(date,symbol,interval)` 拉平算一个相关系数）+ 标准 R² + MSE。
- 因此 **12% 与我们的 0.0776 是同一把尺子**——差距真实，**不是 rank IC / 口径假象**。指标类型上我们没跑偏。

## 2. 原始数据真实长什么样（`data/20230601.h5`，62 列 × 226 interval × 309 票）

| 类别 | 原始列 | 含义 |
|---|---|---|
| **4 档订单簿(LOB)** | `bid0/4/9/19`、`ask0/4/9/19`、`bsize0/asize0`、`bsize0_4/asize0_4`、`bsize5_9/asize5_9`、`bsize10_19/asize10_19`、`btr*/atr*` | 4 档买卖价 + 分档挂单量 + 分档成交额 |
| **逐笔成交流** | `nTradeBuy/Sell`、`tradeBuy/SellQty`、`tradeBuy/SellTurnover`、`tradeBuy/SellHigh/Low`、`buy/sellVwad` | 主动买/卖 笔数·量·额·高低·量价均价 |
| **挂撤单流** | `nAddBuy/Sell`、`addBuy/SellQty/Turnover/High/Low`、`nCxlBuy/Sell`、`cxlBuy/SellQty/Turnover/High/Low` | 挂单(add) + 撤单(cxl)，买卖对称 = 订单流毒性/排队动态 |
| 价/标签 | `midpx`、`lastpx`、`open/high/low`、`fret12` | 中间价、标签(日内 12-interval 前向收益) |

- `interval` 取值 `93000000..144500000` = **09:30:00–14:45:00 的日内 226 个时点**（一条时间序列）。
- 这正是 DeepLOB / 微观结构深度模型专门吃的数据形态。

---

## 3. 我们的 433 特征用了多少、漏了什么（代码证据）

特征 stage（`src/feature_registry.py`）：`base / lag / roll / patch / ofi / trade_impact / conditional_momentum / cross / regime`。

### ✅ 用得不错
- **多档 OFI**（订单流不平衡，0/4/9/19 档 + EMA/sum/z 窗口）：`build_ofi`（registry:737+）；
- **成交不平衡**（trade_imb / turnover_imb / trade_impact 一族）；
- **mid 价时序统计**（roll/patch/lag：rolling mean/std/max/min/slope）；
- **spread、order_pressure**；
- **截面 z / rank**（cross stage，17 个基底列做横截面标准化）。

### ❌ 严重欠用 / 几乎丢弃（这里最可能藏着那 4 个点）
1. **挂撤单流 20 列 → 仅压成 4 个静态比率**：`add_imb / cxl_imb / qty_add_imb / qty_cxl_imb`（`build_base`，registry:651-654）。**没有任何时序、撤单速率、挂撤单与价格变动的交互、队列动态。** 订单流毒性是公认强 alpha 源，这块基本被扔了。
2. **盘口只做成「不平衡摘要」，整个盘口形状没建模**：obi/ofi 把多档盘口压成几个不平衡数，**深度分布、盘口斜率、价档间距的形状演化**没当输入；连 obi 都只到第 9 档（`obi0/obi4/obi9`，最深的 `bsize10_19` 没进 obi）。
3. **成交明细几乎没用**：`buyVwad/sellVwad` 只做了 2 个 gap 特征（registry:655-656）；`tradeBuy/SellHigh/Low` 没用；逐笔成交的量价分布信息丢失。
4. **226 步日内序列被压成固定统计量**：roll/patch/lag 的时序**只在 5 个基础信号上做**（`mid_ret1_raw / obi0 / trade_imb / order_pressure / spread`，registry:696/712/683），而且是人工选的 mean/std/max/min/slope。**全序列的精细演化、其余几十个微观结构通道的时序，DL 都看不到**——因为 DL 喂的是这些摘要，不是原始流。

---

## 4. 这意味着什么（战略重判）

1. **三轮 DL（TCN→GRU→截面）一直≈传统、谁也破不了 0.08 的真因要重判**：不是「数据锁死、可学增量少」，而是**传统和 DL 吃的是同一份被压缩过的 433**；DL 从没机会看原始盘口，自然榨不出更多。`best_epoch=1-2` 更像是「在贫信息的摘要上没什么可学」，不是「数据本身没信息」。
2. **唯一一次喂 raw 的尝试（TCN-on-raw，59 通道）被基于假前提否决**：否决理由写的是「raw 与 433 同源、无更富数据」——现已证伪。且那次失败（0.009）是模型太粗糙（vanilla TCN、4 epoch、无截面、raw 巨量纲没好好归一化），**不是 raw 没 alpha**。
3. **DeepLOB 卡带本来就在规划里**（规格写明「LSTM-on-features → DeepLOB-on-rawLOB 只动两块卡带」），**却因这个假前提从没被真正建过。**

## 5. raw 盘口建模方案（草案，供早上拍板，今晚未写实现）

> 目标：建一个**直接吃原始盘口/订单流序列**的 DL 模型，作为冲 12% 的真入口。脊柱（协议/折/评分）零改，只换 InputAdapter + 卡带。

- **输入**：raw 62 列（或精选微观结构通道）的截面序列张量 `[N票, L步, C通道]` + mask；重点把**目前被丢的挂撤单流、盘口形状、成交明细**全部作为通道喂进去。
- **归一化（关键难点）**：LOB 挂单量/成交额量级巨大且长尾，**必须做 per-symbol / per-day 的滚动标准化或 log 变换**（这正是 TCN-on-raw 当年没做好、导致 0.009 的疑点之一）。沿用脊柱「fit-on-train / apply-on-val」零泄漏归一化。
- **架构候选**：
  - (a) **DeepLOB 式**：CNN 沿「价档维」卷出盘口形状 → Inception/卷积沿时间 → LSTM/TCN 编码日内序列；
  - (b) **复用已有截面卡带**：把它的输入从 433 换成 raw 通道，保留「共享时序腿 + 截面注意力」，让它既看 raw 微观结构、又跨票截面。
- **对齐协议**：用**同一个 `build_dl_folds` 折**（与 P2/传统逐字节同边界）、**pooled Pearson** 评分、训练段 OLS rescale 保 R²——与现有脊柱零冲突。
- **与融合的关系**：raw-DL 若更强、且与传统去相关 → 直接当融合更强的一臂喂进 Tier-1/残差；若它单腿就显著超 0.08 → 主线直接换它。
- **风险**：raw 量纲/归一化难、训练不稳、可能仍欠拟合；**必须 properly normalized + 带截面**，否则重蹈 TCN-on-raw。
- **建议的第一刀（最小验证）**：先只在 **fold2（Nov，传统这里 0.0904）** 上跑一个最小 raw-LOB 原型，看**单折能否 > 0.0904**；过了再铺三折认证，不过就回头查归一化/架构。**先证明 raw 有肉，再投入。**

## 6. 今晚 P2 结果（传统侧地基，已就绪）

`experiments/dump_trad_preds_folds.py` 对**与 DL 完全同边界**的 3 选型折 + 交付折跑锁定融合代表（ridge+lgbm，ridge 已 P1 修 float64）OOS 预测：

| 折 | 窗口 | 自检 Pearson | pred_nan |
|---|---|---|---|
| fold0 | 0831–0927 | 0.0688 | 0 |
| fold1 | 0928–1102 | 0.0696 | 0 |
| fold2 | 1103–1130 | 0.0904 | 0 |
| delivery | 1204–1229 | 0.0770 | 0 |

- **护栏全过**：3 选型折均值 0.0763 / 最坏 0.0688，与 expanding ~0.0776 量级吻合；delivery 0.0770 ≈ Dec sanity 0.0775（P1 后）完全一致（协议搬运无 bug）；**全折 NaN=0**（P1 float64 修复跨所有训练区都稳）。
- 产物：`results/dl/_p2_trad_folds/trad_preds_fold*.csv` + `folds_index.json`（gitignore 不入库，本机可见）。
- **意义**：融合所需的「传统那一列」已在 DL 协议下对齐就绪。但融合（Tier-1 等权）只能榨现有的 ~0.085，**不是冲 12 的路**——12 看第 5 节的 raw 建模。
