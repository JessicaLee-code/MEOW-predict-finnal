更新日期：2026-05-23
状态：**待实施**

## 1. 背景与动机

### 1.1 项目上下文

MEOW-Predict 预测 A 股日内 12 tick 前向收益率（fret12）。后续路线（P1–P5）全部围绕特征工程迭代展开：OFI 动态订单流、成交冲击、条件动量/反转、模型比较、受限融合。

核心迭代循环：**改特征定义 → 重建特征 → 跑 rolling evaluation → 对比 baseline → 再改**。特征管道的效率直接决定实验迭代速度。

### 1.2 当前问题

| 问题 | 表现 | 根因 |
|---|---|---|
| OOM 崩溃 | 4 worker 跑 long_40d_5d 时 3 个 worker 被杀 | 内存 cache 无上限，单 worker 峰值 18-20 GB |
| 全量重算浪费 | 改一个特征组要重算 144 天 × 全部 9 组 ≈ 17 min | build() 是 monolithic 线性链，不支持按组重建 |
| 假依赖链 | build() 把全列 DataFrame 逐层传递，暗示线性依赖 | 实际依赖是 DAG，ofi 完全不依赖 base |
| ExperimentRunner 过载 | 900 行 god object，缓存/加载/训练/评估全揉一起 | 职责未分离 |
| 加新特征组要改两处 | 写 builder + 手动更新 select_groups dict | 两处信息未关联 |

### 1.3 设计宪法

> 系统不是为了做漂亮的 feature store，而是为了支撑冲高分实验的快速迭代、自由试错、计算复用和特征生命周期管理。

一票否决规则——以下任一条成立则方案有问题：

- 为了试一个新特征，需要手动改多个无关模块
- 改一个特征组导致全量 144 天所有组无脑重算
- 缓存失效规则不清楚，结果不知来自新代码还是旧缓存
- 最终 meow.py 离开本地 cache 就跑不起来

### 1.4 不在本次范围

- 深度学习特征（序列特征、注意力特征）
- 跨日特征（隔夜缺口等，当前所有特征均为日内）
- YAML/JSON 配置系统
- 外部框架引入（Hamilton 等留作后续评估）
- Trainer / EvalProtocol / Scheduler 的接口变更

## 2. 目标架构

### 2.1 总体分层

```
┌──────────────────────────────────────────────────────────┐
│  实验层（不变）                                            │
│  ALL_SPECS / RIDGE_SPECS + EvalProtocol + Scheduler       │
│  实验配置仍为 Python dict，不引入新格式                      │
└──────────────────────┬───────────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────────┐
│  特征层（本次重构核心）                                     │
│                                                          │
│  FeatureRegistry    特征组定义、依赖声明、group 映射        │
│  FeatureStore       磁盘缓存管理、脏检测、拓扑重建          │
│  FeatureLoader      按需读取 parquet、选列、concat          │
└──────────────────────┬───────────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────────┐
│  模型层（瘦身后的 ExperimentRunner）                        │
│  fit_model / predict / evaluate（不再负责数据加载和缓存）    │
└──────────────────────────────────────────────────────────┘
```

### 2.2 数据流（改造后）

```
[预构建阶段 — 一次性或特征变更后执行]

  data/*.h5 (原始 tick)
      │
      ▼
  FeatureStore.build()
      │  按 FeatureRegistry 的拓扑序逐组构建
      │  每组只接收 raw + 已声明的上游组输出（不传冗余列）
      │  每组每天输出一个 parquet 文件
      ▼
  data/features/
      base/20230601.parquet
      ofi/20230601.parquet
      ...
      manifest.json


[实验运行阶段]

  Scheduler worker
      │
      ▼
  FeatureLoader.load(dates, groups=["legacy", "norm_core"])
      │  1. 查 registry 确定需要哪些 stage (base + cross)
      │  2. 读对应 stage 的 parquet 文件
      │  3. 选列（只取 group 声明的列）
      │  4. concat 多天
      ▼
  xdf, ydf（内存中，用完即释放）
      │
      ▼
  ExperimentRunner.fit_model → predict → evaluate
```

### 3.1 FeatureRegistry

**职责**：管理所有特征组的定义、依赖关系、输出列归属。

**文件**：`src/feature_registry.py`（新建）

#### 3.1.2 依赖 DAG（当前 9 组的实际依赖关系）

```
raw ─────┬── base ─────┬── lag
         │             ├── roll
         │             ├── patch
         │             └── regime
         │
         ├── ofi                          (独立，不依赖 base)
         │
         ├── trade_impact                 (deps: base, ofi)
         │
         ├── cross                        (deps: base, ofi, trade_impact)
         │
         └── conditional_momentum         (deps: base, ofi, trade_impact)
```

注意：当前代码中 trade_impact 对 ofi 是软依赖（`df.get("ofi_total", 0.0)`），
重构后改为硬依赖（函数参数显式接收 ofi DataFrame）。

#### 3.1.3 select_groups 自动派生

不再维护独立的 select_groups dict。查询流程：

```python
registry.resolve_groups(["legacy", "norm_core"])
# → 遍历所有 stage 的 groups 声明
# → 找到哪些 stage 包含 "legacy" 或 "norm_core"
# → 返回: {需要加载的 stage 集合, 每个 stage 中要选的列}
```

**特殊情况**：`"norm_core"` 横跨 base 列 + cross 的 `_cs_z` 列。处理方式：

```python
@registry.stage(
    name="cross",
    deps=["base", "ofi", "trade_impact"],
    groups=["cross_z", "cross_rank", "cross_rank_features",
            "norm_core",    # cross 的 _cs_z 列也属于 norm_core
            "cross"],
)
```

一个 group name 可以出现在多个 stage 的 groups 列表中。loader 加载时取并集。

#### 3.1.4 Group 内的列过滤（已定）

**决策**：group_columns 中的精确列名由 build 后扫描实际 parquet 输出自动生成，写入 manifest.json 的 `columns` 字段。

对于需要精细区分子集的 group（如 `"legacy"` 只取 base 的部分列），在 registry 中声明精确列名：

```python
@registry.stage(
    name="base",
    deps=[],
    groups=["legacy", "base"],
    status="promoted",
    group_columns={
        # 精确列名子集（可选，不声明则该 group 取本 stage 全部列）
        "legacy": ["ob_imb0", "ob_imb4", "ob_imb9",
                    "trade_imb", "trade_imbema5", "lagret12"],
    },
)
```

对于动态列名的 group（如 `"ofi_dynamic"` 的 `_ema`/`_sum` 系列列），不在 registry 中声明 pattern。
而是在首次 build 完成后，从 parquet 扫描实际输出列写入 manifest。

`resolve_groups()` 的查询流程：
1. 检查 group_columns 是否有该 group 的精确声明 → 有则用声明的列名
2. 否则从 manifest 读取该 stage 的全部列名作为该 group 的列集合

未在 group_columns 中声明的 group 默认取本 stage 的全部输出列。

#### 3.1.5 Registry 提供的方法

```python
class FeatureRegistry:
    def stage(name, deps, groups, group_columns=None, status="promoted")
        # 装饰器，注册一个 stage
        # status: "promoted"（基线稳定使用）/ "candidate"（验证中）/ "archived"（废弃）

    def topo_order(include_archived=False) -> List[str]
        # 返回拓扑排序后的 stage 名称列表
        # 默认排除 archived stage

    def downstream(stage_name) -> Set[str]
        # 返回指定 stage 的所有下游 stage（传递闭包）

    def resolve_groups(group_names) -> Dict[str, List[str]]
        # 输入 group 名称列表
        # 返回 {stage_name: [需要选取的列名]} 映射
        # 自动排除 archived stage

    def get_builder(stage_name) -> Callable
        # 返回 builder 函数引用

    def get_deps(stage_name) -> List[str]
        # 返回直接依赖的 stage 名称

    def get_status(stage_name) -> str
        # 返回 stage 状态

    def code_hash(stage_name) -> str
        # 返回 builder 函数源码的 md5 hash
        # 用于脏检测
```

#### 3.2.3 脏检测逻辑

```python
FEATURE_COMMON_VERSION = "common-v1"

def find_dirty_stages() -> Set[str]:
    """
    脏检测三层逻辑（任一触发即标脏）：

    1. common_version 不一致 → 全部 stage 标脏
       （公共工具函数语义变更时手动递增 common-v1 → common-v2）
    2. 某 stage 的 code_hash 与 manifest 不一致 → 该 stage + 全部下游标脏
    3. manifest 中不存在某 stage → 该 stage 标脏

    注意：code_hash 基于 inspect.getsource(builder_func)。
    如果 builder 调用的公共工具函数（如 _safe_div）发生变更，
    builder 自身 hash 不变，需要通过递增 FEATURE_COMMON_VERSION 触发重建。
    """
```

#### 3.2.4 构建流程

```python
def build(dates, force=False, stages=None):
    """
    构建特征并写入磁盘。

    参数:
        dates:  要构建的交易日列表
        force:  True 时忽略 hash 检测，全部重建
        stages: 指定只重建某些 stage（调试用），None 表示按脏检测决定

    流程:
        1. find_dirty_stages() 确定需要重建的 stage 集合
        2. 按 registry.topo_order() 顺序，只执行脏 stage
        3. 对每个脏 stage、每一天：
           a. 读取 raw h5 数据
           b. 读取该 stage 依赖的上游 stage 的当天 parquet（已构建）
           c. 调用 builder(raw, **upstream_outputs) → DataFrame
           d. 写入 parquet
        4. 更新 manifest.json（写入新 hash + 时间戳）
    """
```

#### 3.2.5 命令行入口

```bash

```

### 3.3 FeatureLoader

**职责**：从磁盘读取已构建的特征，按 group 选列，concat 多天，返回 xdf/ydf。

**文件**：`src/feature_loader.py`（新建）

#### 3.3.1 主接口

```python
class FeatureLoader:
    def __init__(self, feature_dir: str, registry: FeatureRegistry):
        self.feature_dir = feature_dir
        self.registry = registry

    def load(self, dates: tuple, groups: list) -> Tuple[pd.DataFrame, pd.DataFrame]:
        """
        加载指定日期范围 + 特征组的数据。

        流程:
            1. registry.resolve_groups(groups)
               → 确定需要哪些 stage + 每个 stage 选哪些列
            2. 对每一天：
               a. 读取所需 stage 的 parquet
               b. 按 group_columns 选列
               c. 横向拼接各 stage 的列 + meta 列 + target 列
            3. 纵向 concat 所有天
            4. 返回 (xdf, ydf)

        特性:
            - 无状态，不持有任何缓存
            - 多进程安全（每个 worker 独立读文件）
            - 返回后 caller 持有数据的唯一引用，用完可释放
        """
```

#### 3.3.2 meta 列和 target 列的处理（已定）

**决策**：FeatureLoader 直接从 raw h5 读取 meta 列（date, symbol, interval）和 target 列（fret12）。

stage parquet 纯粹只存特征列，不含 meta/target。这样 stage 的职责干净——只管特征计算。
h5 在 OS page cache 中，读取几个 meta 列的成本极低（< 0.1s/天）。

```python
# FeatureLoader.load() 内部流程：
# 1. 从各 stage parquet 读取特征列
# 2. 从 raw h5 读取 meta + target 列
# 3. 横向拼接（共享相同的行顺序，无需 join）
# 4. 纵向 concat 多天
```

#### 3.3.3 多 stage 拼接时的 key 对齐校验

loader 横向拼接多个 stage 的 parquet 时，**必须校验行对齐**：

```python
# 每天拼接时，assert 所有 stage 的 (date, symbol, interval) key 与 h5 原始顺序一致
# 不一致直接报错，不默默 concat
assert len(stage_df) == len(meta_df), f"行数不匹配: {stage_name}"
assert (stage_df.index == meta_df.index).all(), f"行顺序不匹配: {stage_name}"
```

这防止某个 stage 的 builder 意外改变了行顺序，导致特征和 target 错位——这类 bug 不会报错但会污染所有模型结果，极难排查。

### 3.5 Scheduler 调整

#### 3.5.1 Worker 函数变更

```python
def _fold_group_worker(args: tuple) -> List[dict]:
    h5dir, fold_group, specs, completed_keys, feature_dir = args

    # 不再创建 ExperimentRunner 实例来做数据加载
    # 创建 FeatureLoader（无状态，无缓存）
    loader = FeatureLoader(feature_dir, registry)

    # 创建瘦身后的 runner（只做训练和评估）
    runner = ModelRunner(...)

    for meta in fold_group.fold_metas:
        for spec in specs:
            # loader 从 parquet 读取数据
            xtrain, ytrain = loader.load(meta.train_dates, groups=spec["groups"])
            xval, yval = loader.load(meta.val_dates, groups=spec["groups"])

            # runner 只做训练和评估
            result = runner.train_and_evaluate(xtrain, ytrain, xval, yval, spec)
            results.append(result)

            # 用完立即释放，不缓存
            del xtrain, ytrain, xval, yval

    return results
```

### 3.6 meow.py 提交入口约束

最终提交的 `meow.py` **不依赖** `data/features/` 目录。

```python
# meow.py 的特征构建路径:
# 直接使用 FeatureRegistry 的 builder 函数，从 raw h5 实时构建
# 与 FeatureStore.build() 调用的是同一套代码

raw = loader.loadDate(date)
features = {}
for stage_name in registry.topo_order():
    builder = registry.get_builder(stage_name)
    deps = {dep: features[dep] for dep in registry.get_deps(stage_name)}
    features[stage_name] = builder(raw, **deps)
```

缓存（data/features/）是开发加速层。builder 函数是真相源。
两条路径用同一份代码，保证一致性。

## 6. 已知限制与后续扩展

### 6.1 已知限制

- **公共工具函数变更需手动递增 `FEATURE_COMMON_VERSION`**：`code_hash` 基于 builder 函数自身源码。如果修改了 `_safe_div` 等被多个 builder 调用的工具函数，需手动递增 `FEATURE_COMMON_VERSION`（如 `common-v1` → `common-v2`）触发全量重建。比 `--force` 更可追溯
- **日内特征假设**：所有 builder 函数假设输入是单天数据，不支持跨日特征。如需隔夜缺口等跨日特征，需要扩展 build 流程的输入模式
- **单机串行构建**：build 阶段逐天串行处理。144 天 × 9 组约 17 分钟，当前可接受。如组数增长到 30+，可考虑天级并行

### 6.2 后续扩展点

- **交互项探索**：在实验脚本中用 pandas 手动计算（`df["a"] * df["b"]`），验证有信号后提升为正式 stage
- **特征生命周期**：stage 装饰器可添加 `status` 字段（scratch / candidate / promoted / archived），目前用注释管理即可
- **Hamilton 迁移**：registry 接口稳定后，可评估将底层替换为 Hamilton DAG executor，builder 函数签名基本兼容
- **跨日特征**：扩展 builder 签名，支持接收前 N 天的已构建特征作为额外输入
