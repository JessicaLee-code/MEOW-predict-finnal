## 1. 设计目标

### 解决的问题

| 现状痛点 | 设计目标 |
|---|---|
| 三层串行 for 循环，跑 `--suite ridge` 需数小时 | 并发执行，目标压缩至 30 分钟内 |
| 无断点续跑，中断后全部重来 | 按 job 落盘，重跑自动跳过已完成 |
| 新实验类型（模型/特征/目标变换）需改 runner 主体 | Trainer 接口隔离，新增方法只加一个类 |
| DL 接入无自然插入点 | 预留 `BaseTrainer` 接口，未来一个子类搞定 |

### 不在本次实施范围

- 深度学习 Trainer 实现（接口预留，代码不写）
- 多 seed 管理（固定 seed=42）
- YAML/JSON 配置文件系统
- 实验注册中心 / auto-discovery
- 分布式执行

## 3. 关键数据结构

### FoldData

worker 进程内构造，不跨进程传递（只传元数据）。

```python
@dataclass
class FoldData:
    profile_name: str
    fold_id: int
    train_dates: tuple   # 交易日 int 元组
    val_dates: tuple
    # 以下字段在 worker 内由 ExperimentRunner.load_feature_split() 填充
    # 不序列化，不跨进程传
    xtrain: Optional[pd.DataFrame] = None
    ytrain: Optional[pd.DataFrame] = None
    xval:   Optional[pd.DataFrame] = None
    yval:   Optional[pd.DataFrame] = None
```

跨进程只传 `FoldDataMeta`（轻量可 pickle）：

```python
@dataclass(frozen=True)
class FoldDataMeta:
    profile_name: str
    fold_id: int
    train_dates: tuple
    val_dates: tuple
```

### FoldResult

所有 Trainer 统一返回格式，ResultStore 只认这个结构。

```python
@dataclass
class FoldResult:
    profile_name: str
    fold_id: int
    experiment_id: str
    feature_set: str
    model_type: str
    target_type: str
    postprocess_type: str
    train_corr: float
    val_corr: float
    train_mse: float
    val_mse: float
    train_r2: float
    val_r2: float
    daily_corr_mean: float
    daily_corr_std: float
    train_val_corr_gap: float
    runtime_sec: float
    status: str        # "ok" | "error"
    error_msg: str     # status=error 时填写
    notes: str
```

### BaseTrainer 接口

DL 唯一需要实现的接口。

```python
class BaseTrainer(ABC):
    def __init__(self, spec: dict):
        self.spec = spec

    @abstractmethod
    def run_fold(self, fold_data: FoldData) -> FoldResult:
        """
        输入已加载的 train/val 数据，输出统一 FoldResult。
        实现负责：特征过滤、模型训练、预测、指标计算。
        不负责：数据加载（由调度层完成）、结果落盘（由 ResultStore 完成）。
        """
        ...
```

**未来加 DL 时只需**：

```python
class SequenceTrainer(BaseTrainer):
    def run_fold(self, fold_data: FoldData) -> FoldResult:
        # 处理 epoch / early stopping / GPU checkpoint
        # 可读取 fold_data.xtrain 直接用，或自行从 raw 构建序列
        ...
```

其余层（调度、协议、结果）一字不动。

## 4. 并发调度设计

### 任务展开策略

目标：充分利用 M5 MacBook Air 的 10 核（4 P 核 + 6 E 核），推荐 **4 个 worker 进程**（16 GB 内存限制，详见下方内存分析）。

关键约束：**同一 profile 内相邻 fold 的 train_dates 高度重叠**（如 medium_20d_5d 相邻 fold 重叠 ~60%），若相邻 fold 分配到同一 worker，进程内 `_daily_feature_cache` 可大量复用，节省特征重建时间。

展开方案：**按 profile 把 fold 列表切成若干连续子组（fold-group），每个子组作为一个 worker 任务**。

```
short_8d_2d    (19 folds) → 3 个子组：fold 0-6 / 7-12 / 13-18
medium_20d_5d  (16 folds) → 3 个子组：fold 0-5 / 6-10 / 11-15
long_40d_5d    ( 6 folds) → 1 个子组：fold 0-5
expanding_40d_5d(12 folds)→ 2 个子组：fold 0-5 / 6-11
──────────────────────────────────────────
总计：9 个子组 → 9 个 worker 任务 → ≤8 并发
```

子组内按 fold 顺序串行执行，充分利用进程内 cache。子组间完全并行。

### Worker 函数（模块级，可 pickle）

```python
def _fold_group_worker(args: tuple) -> list[FoldResult]:
    """
    进程池 worker。接收一批连续 fold 的元数据，
    在进程内独立创建 ExperimentRunner，顺序执行各 fold 的所有 spec。
    """
    h5dir, profile_name, fold_metas, specs = args
    runner = ExperimentRunner(h5dir)       # 进程内独立 cache
    results = []
    for meta in fold_metas:
        fold_data = FoldData(...)
        for spec in specs:
            trainer = TabularTrainer(spec, runner)
            result = trainer.run_fold(fold_data)
            results.append(result)
        # 每折结束清空 split_cache，防止跨折积累巨型 DataFrame（OOM 修复）
        # _daily_feature_cache 保留，避免相邻折重复磁盘 IO
        runner._split_cache.clear()
        runner._raw_split_cache.clear()
        gc.collect()
    return results
```

### 内存分析（M5 MacBook Air 16 GB）

| 缓存层 | 职责 | 生命周期 |
|---|---|---|
| `_daily_feature_cache` | 每天特征/目标 DataFrame（~44 MB/天） | worker 进程内全局，跨折复用 |
| `_split_cache` | concat 后的完整训练/验证集 | **每折结束后主动清空**（OOM 修复） |

**OOM 根因**（2026-05-22 实测）：`_split_cache` 按 `(train_dates_tuple, max_days)` 缓存，rolling fold 场景下每折 key 唯一，等于每折都多存一份全量 concat DataFrame 且永不释放。`expanding_40d_5d` 末尾单条 entry ≈ 4 GB，7 折一组积累 >20 GB；8 workers 并行后总峰值超 50 GB，触发 macOS OOM 重启。

**修复后峰值估算（n_workers=4）**：
- 每 worker：`_daily_feature_cache`（~2–5 GB）+ 当前折 split（~1–4 GB）≈ 6–9 GB
- 4 workers 同时运行：~25–35 GB（含 swap），16 GB 物理内存可接受

### ParallelScheduler 主逻辑

```python
class ParallelScheduler:
    def __init__(self, h5dir, n_workers=4, output_dir=None):  # 16 GB Mac 推荐 4
        ...

    def run(self, specs, profiles, fold_groups, resume=True):
        # 1. 如果 resume=True，从 fold_metrics.csv 读取已完成的 job key
        completed = self._load_completed_jobs() if resume else set()

        # 2. 过滤掉已完成的 fold-group（按 profile+fold_range 粒度）
        pending = [g for g in fold_groups if not self._is_completed(g, completed)]

        # 3. 进程池并发执行
        with ProcessPoolExecutor(max_workers=self.n_workers) as pool:
            futures = {pool.submit(_fold_group_worker, (self.h5dir, g.profile_name,
                       g.fold_metas, specs)): g for g in pending}
            for future in as_completed(futures):
                results = future.result()
                self._append_results(results)   # 实时落盘
```

## 5. Resume 机制

**落盘粒度**：每个 `(profile_name, fold_id, experiment_id)` 组合落一行到 `fold_metrics.csv`。

**Resume 判断**：启动时读取已有 `fold_metrics.csv`，构建 `completed_keys` set，跳过已存在的 job。

**异常处理**：worker 抛异常时，`FoldResult.status="error"` 落盘，不影响其他 job 继续执行。下次 resume 时 error job 也会重跑（status != "ok" 不视为完成）。

## 6. DL 扩展点说明

将来接入深度学习时，**只需以下改动，不触碰现有逻辑**：

1. 实现 `SequenceTrainer(BaseTrainer)`，在 `run_fold()` 内处理：
   - Epoch 训练 / early stopping
   - GPU 资源分配（通过环境变量或 `spec` 字段传入）
   - Checkpoint 保存/恢复
   - 序列数据构建（如果需要从 `fold_data.xtrain` 重整为 3D tensor）

2. 在 `ALL_SPECS` 里加入 DL spec，type 字段设为 `"dl_sequence"` 或类似值。

3. `_fold_group_worker` 中的 Trainer 路由逻辑加一个 `elif`：
   ```python
   elif spec["type"].startswith("dl_"):
       trainer = SequenceTrainer(spec)
   ```

4. 如果 DL 需要 GPU 且不适合 ProcessPoolExecutor（多进程争 GPU），可单独为 DL job 用 `ThreadPoolExecutor` 或串行模式，传统 ML job 仍用进程池。
