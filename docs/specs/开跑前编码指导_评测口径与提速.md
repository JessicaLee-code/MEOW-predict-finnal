# 开跑前编码指导：两速评测口径落地与 expanding 提速

创建日期：2026-05-25
适用对象：coding agent（按本文件改代码；顾问与复审 agent 不直接改代码）

## 这份文档是什么

评测协议本轮收敛为**两速结构**（快车道 short+long+每日IC 筛选 / 慢车道 expanding+Review 关口）。
本文件把"连带要改的工程项"列成可执行清单，精确到文件与验收。

- **规则与"为什么"不在这里**：判定逻辑见 `AGENTS.md §4.2 / §4.4 / §4.6 / §7.11`，背景白话见 `NOTE.md`《最终落定：两速评测结构》《基础设施这次要改什么、为什么》。
- 本文件只写"改哪里、改成什么、怎么验收"。涉及的文件名若与实际不符，以代码实际为准，按意图落地即可。

## 总原则（不可违背）

1. **不改训练 / 评测协议主体逻辑**：本轮只动 suite 默认、profile 分层、调度成本、输出诊断、数据 dtype，以及 make_decision 的判定阈值。fold 构造、泄漏防护、target 还原口径一律不动。
2. **每一项都要有验收手段**（单测或一次对照跑），尤其 float32 这种可能影响数值的改动。
3. 改动按 commit 粒度拆开（AGENTS §9），一条一提交，不要混提。

## 1. 日常 suite 默认 = short + long（medium 移出日常）

- **文件**：`experiments/p0_eval_protocol.py`（suite 定义处）、必要时 `src/eval_protocol.py` 的 profile 选择。
- **改什么**：
  - 新增（或调整）一个"日常筛选" suite，默认 `profiles = [short_8d_2d, long_40d_5d]`。
  - `medium_20d_5d` 不进日常默认；保留 `--profiles medium_20d_5d` 可单独 ad-hoc 跑。
  - `expanding_40d_5d` 不进日常默认（它只在关口跑，见第 2 条）。
- **为什么**：AGENTS §4.2。short 出统计功效，long 出"长窗口斜率/否决票"，medium 是冗余中间插值。
- **验收**：日常 suite 跑一轮，确认只跑 short+long、且 ~10–15 min 量级。

## 2. expanding 关口跑法：候选 vs 基线 + float32 + 2 worker

expanding 是 promote 硬门槛，只在提交关口跑，要又快又省。

- **2a. 只跑候选 vs 基线**
  - **文件**：`experiments/p0_eval_protocol.py`（关口入口 / spec 选择）。
  - **改什么**：提供一个"关口"跑法，只评候选 spec + 当前基线 spec（2 个），不跑 R00–R04 全套。
  - **为什么**：AGENTS §4.4。关口是"这个候选要不要并入 backbone"，不是重建排行榜。

- **2b. 特征 dtype 改 float32**
  - **文件**：`src/feature_loader.py`（加载后 / 返回前），或特征落盘处。
  - **改什么**：特征矩阵以 float32 进入训练（标签按需）。
  - **为什么**：内存砍半 + 单精度 BLAS 更快。
  - **验收（强制）**：同一 spec 同一 fold，float32 与 float64 的 `protocol_corr` 差异在数值噪声内（建议 |Δcorr| < 1e-4）。**不达标不许合入。** 注：这是同折逐点数值一致性检查，和 §4.3 的采纳门槛（≥0.003）、标准误（≈0.006）是两码事，别混用。

- **2c. expanding 批次 worker 提到 2 + 成本均衡切分**
  - **文件**：`src/scheduler.py`（heavy 批次并发上限、group 切分）。
  - **改什么**：
    - 代码层把 heavy 批次（含 expanding）并发上限做成可由 1 提到 2 的能力（float32 数值对照已于 #16 验收通过，`|Δcorr|=0`）。
    - group 切分不要用连续 `[fold0..5][fold6..11]`（会把长窗口重折全堆一个 worker）；按预估成本（训练窗口长度 ∝ fold 序号）均衡分到两个 worker，使两个 worker 负载相当、且峰值期不撞车。
  - **为什么**：NOTE《基础设施》。expanding 后段 fold 训练窗口最长、最重，要打散。
  - **运行口径（2026-05-26 锁定，权威）**：
    - **macOS（当前 16GB 机器）正式按单 worker 串行跑 expanding 关口（`--n-workers 1`），不开双并行。** 依据：最重单组实测峰值 RSS `8.34 GB` 是单折载入瞬时尖峰、稳态仅 ~3 GB，串行在 16GB Mac 上稳；而双并行尾段（`95+90` 天）瞬时峰理论 ~17–18 GB 超物理内存有 OOM 风险。expanding 是低频提交关口，串行多花的墙钟可接受，换确定性更划算。
    - **2 worker 并行仅在 20GB+ 空闲内存机器（如 Windows）启用**：成本均衡切组（非连续）+ 单测（`tests/test_scheduler.py`）已就绪，迁机后直接开即可，余量充足。
  - **验收**：① 单测锁“非连续切组 + 成本接近”；② macOS 串行下 expanding 关口跑（2 spec）完成、无 OOM、无 worker 异常退出，RSS 看门狗不触发硬阈值。

## 3. 明确不做：增量正规方程

- **不实现**"维护滚动充分统计量、每折只加增量"的增量 Ridge。
- **理由**：它只对线性闭式解有效、P4 上树即失效；而 expanding 是偶发关口、第 2 条的便宜招已够。投入产出不划算。
- 记录在此是为了**防止有人又去造它**。若将来 expanding 变成高频主回路再重新评估。

## 4. make_decision 硬契约 + 单测

- **文件**：`src/eval_protocol.py`（`make_decision` 或等价判定函数）+ 新增测试文件。
- **改什么**：严格按 AGENTS §4.6 判定表返回 `reject / review / promote`：
  - `delta_corr < 0.003` → `reject`
  - `0.003 ≤ delta_corr < 0.005` → `review`（禁止直接 promote）
  - `delta_corr ≥ 0.005` 且 expanding 已跑且不为负 且 short 与 expanding 同向为正、long 不显著翻负 且 每日 IC 不恶化 且 stability 不降 且 无新强负折 → `promote`
  - 其它 → `review`
  - **缺 expanding 结果时，最高只能 `review`，不得 `promote`。**
- **排行榜排序键**：头条按 `protocol_corr_mean` 排，`stability_score` 作并排守门指标（不再当第一排序键）。
- **验收（强制单测，至少 3 条断言）**：
  1. `delta_corr=0.004` 必须返回 `review`；
  2. 缺 expanding 必须不能 `promote`；
  3. 出现新的强负折必须不能 `promote`。

## 5. 每日 IC / IC-IR 提为一等输出

- **文件**：`src/eval_protocol.py`（fold_metrics / profile_summary / leaderboard 组装处）；每日截面 IC 可能已在 `src/eval.py` 算。
- **改什么**：把每日截面 IC 序列的均值、以及"均值 ÷ 波动"（IC-IR）并进 leaderboard 主视图，与池化 corr 并列展示。
- **为什么**：AGENTS §4.7 第 4–5 步 / NOTE。用来分辨"大盘的钱 vs 选股的钱"；代码多半已在算，零额外计算成本，只是没并进主视图。
- **验收**：leaderboard 输出含每日 IC 均值与 IC-IR 两列。

## 6. 训练目标 winsorize

- **文件**：`src/experiment_runner.py`（构造 `ytrain` 处，紧邻 `fit_model` 的 StandardScaler 前）。
- **改什么**：只裁训练标签 `ytrain`，按**训练集**分位数两侧裁剪（非对称）；测试 / 提交永远输出原始 `fret12`，不做逆变换。
  - clip 候选 `{P0.5/P99.5, P1/P99, 不裁}`，与 alpha 在 P0.5 同一次 short+medium 轻扫里一起定，落平台区锁死。
- **为什么**：AGENTS §7.11 / NOTE。它改的是训练系数（→ 改预测 → 改池化 Pearson），不是"只稳梯度"；对 §7.5 target 还原无影响、不构成泄漏。
- **验收**：winsorize 开关 + 分位参数可配；锁定后 P1–P3 不逐 spec 调。

## 7. 长跑别让 Mac 睡

- **做法**：长任务（expanding 关口、全量构建）跑前挂 `caffeinate -i -s`，或用 `caffeinate -i <命令>` 包裹；可与 `experiments/run_with_memory_guard.py` 叠加。
- **为什么**：本轮 expanding 跑因机器休眠多停了约半小时，且污染计时。
- **验收**：长跑期间机器不进睡眠，耗时统计连续。
